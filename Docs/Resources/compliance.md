# Compliance Documentation and Resources

## What This Page Is For

This page is a curated reference for compliance, assurance, security control frameworks, secure development standards, and supply-chain security frameworks.

It is meant to make it easier to find the right official documentation when working on:

- privacy and data protection
- healthcare security and privacy
- audit and assurance readiness
- control framework mapping
- secure software development
- cloud security alignment
- supply-chain and repository security

## How To Use This Page

Not everything here is the same kind of thing.

Some items are:

- laws or regulations
- audit and assurance report frameworks
- control catalogs
- security maturity models
- secure development standards
- supply-chain security frameworks
- repository or open-source health signals

Use this page as a map, not as a claim that all of these frameworks should be treated the same way.

## Privacy, Healthcare, and Payment Standards

### GDPR

The General Data Protection Regulation is the core EU privacy and personal data protection regulation.

- [GDPR Official Text (EUR-Lex)](https://eur-lex.europa.eu/eli/reg/2016/679/oj/eng)

### HIPAA

HIPAA includes privacy and security requirements for protected health information in regulated healthcare contexts.

- [HIPAA Security Rule Summary](https://www.hhs.gov/hipaa/for-professionals/security/laws-regulations/index.html)
- [HIPAA Privacy Rule Summary](https://www.hhs.gov/hipaa/for-professionals/privacy/laws-regulations/index.html)
- [HIPAA Security Rule Guidance Material](https://www.hhs.gov/hipaa/for-professionals/security/guidance/index.html)

### PCI DSS

PCI DSS is the core baseline for protecting payment account data.

- [PCI DSS Standard Overview](https://www.pcisecuritystandards.org/standards/pci-dss/)
- [PCI Security Standards Council](https://www.pcisecuritystandards.org/)
- [PCI Security Documents Library](https://www.pcisecuritystandards.org/document_library/)

## Audit, Assurance, and Trust Frameworks

### TSC

The Trust Services Criteria define the control criteria behind SOC reporting for security, availability, processing integrity, confidentiality, and privacy.

- [2017 Trust Services Criteria with Revised Points of Focus](https://www.aicpa-cima.com/resources/download/2017-trust-services-criteria-with-revised-points-of-focus-2022)

### SOC 2

SOC 2 is an examination of controls at a service organization against the Trust Services Criteria.

- [SOC 2 Overview](https://www.aicpa-cima.com/topic/audit-assurance/audit-and-assurance-greater-than-soc-2)
- [SOC Suite of Services Overview](https://www.aicpa-cima.com/resources/landing/system-and-organization-controls-soc-suite-of-services)

### SOC 3

SOC 3 is the general-use version of reporting against the Trust Services Criteria.

- [SOC 3 Overview](https://www.aicpa-cima.com/topic/audit-assurance/audit-and-assurance-greater-than-soc-3)
- [Illustrative SOC 3 Report](https://www.aicpa-cima.com/resources/download/illustrative-soc-3-r-report)

11. SLSA
12. OpenSSF Scorecard

## How These Fit Together

A useful way to think about these resources:

- **GDPR, HIPAA, and PCI DSS** are regulatory or industry obligations
- **TSC, SOC 2, and SOC 3** are assurance and reporting structures
- **NIST SP 800-53, NIST CSF, ISO 27001, ISO 27002, CIS Controls, and CSA CCM** are control or risk-management frameworks
- **OWASP ASVS, OWASP SAMM, and NIST SSDF** are software and application security guidance
- **OWASP Top 10** is a risk-awareness lens
- **SLSA and OpenSSF Scorecard** are supply-chain and repository-security lenses
- **CAIQ** is best understood as a questionnaire and evidence mechanism aligned to **CCM/STAR**, not a standalone primary control framework

## Notes

A few practical habits make compliance resources more useful:

- start by identifying whether you need a law, a control framework, an assurance framework, or an engineering standard
- do not treat repository-level evidence as proof of full organizational compliance
- use mappings and crosswalks when you need to align multiple frameworks
- treat secure development and supply-chain guidance as part of compliance readiness, not something separate
