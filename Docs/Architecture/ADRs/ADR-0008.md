## Context

The platform requires **progressive delivery** across Kubernetes clusters and within each cluster:

* safer rollouts with gradual traffic shifting
* automated analysis of health signals
* fast rollback on regression
* standardized rollout patterns across environments

The promotion model is **GitOps**:

* application changes are promoted through Development → Staging → Production via PRs (ADR-0007)

But within a given cluster, deployments still need safe rollout behavior. A purely “replace all pods” deployment approach increases risk.

Additionally, the platform has a **Zero Trust** posture requiring mTLS for east/west service traffic. A service mesh provides both security and traffic-management primitives that are useful for progressive delivery.

Observability is mandated (ADR-0014). Progressive delivery requires reliable telemetry signals to make automated decisions.

## Decision

Adopt **Istio** as the service mesh and **Flagger** as the progressive delivery controller for **in-cluster canary deployments**, integrated into FluxCD GitOps workflows.

Key elements:

* **Istio** provides:

  * in-cluster **mTLS** for service-to-service communication
  * traffic shaping (weighted routing)
  * telemetry integration points

* **Flagger** provides:

  * automated canary analysis driven by metrics/logs
  * traffic shifting and rollback using Istio routing
  * declarative canary definitions stored in Git

This decision is specifically for **in-cluster** canaries.

Cross-cluster promotion remains GitOps-driven:

* The platform does not “teleport pods between clusters.”
* Workloads are promoted by merging Git changes that update the desired state in the next environment.

## Scope

This ADR governs:

* the standard in-cluster progressive delivery mechanism
* the selection of Istio and Flagger
* the expectation that canaries are declared in Git and reconciled by Flux

This ADR does not:

* define the full ingress architecture (gateway placement and public routing)
* define final SLO thresholds and default canary templates (these are implemented as code and may evolve)
* replace environment promotion policy (dev → staging → prod remains Git-based)

## Alternatives Considered

### A1) No progressive delivery (rolling updates only)

**Pros**

* simplest

**Cons**

* higher risk rollouts
* manual detection and rollback
* inconsistent outcomes across environments

### A2) Argo Rollouts

**Pros**

* strong progressive delivery features

**Cons**

* less aligned with Flux-first model
* additional controller and operational model to manage

### A3) Ingress-controller canaries (NGINX/Traefik) without a mesh

**Pros**

* less control-plane overhead than a full mesh

**Cons**

* canary control limited to ingress traffic
* does not address east/west service mTLS
* less consistent for internal-only services

### A4) Linkerd + Flagger

**Pros**

* lighter mesh footprint
* simpler operational profile

**Cons**

* Istio chosen due to broader feature set and integration patterns
* may require different traffic management primitives depending on needs

## Rationale

Istio + Flagger is selected because it:

1. **Unifies security and rollout controls**

   * mTLS and traffic shifting are managed in one mesh layer.

2. **Enables automated, metrics-driven canaries**

   * Flagger uses telemetry to decide promotion/rollback.

3. **Fits GitOps and compliance expectations**

   * Canary configuration is declared in Git.
   * Changes are reviewable and auditable.

4. **Supports consistent patterns**

   * Standard canary templates and SLO checks can be reused across workloads.

5. **Scales to complexity over time**

   * Can start with limited namespaces and expand once stable.

## Consequences

### Positive

* safer deployments and faster regression detection
* consistent rollback behavior
* mesh mTLS strengthens zero-trust posture

### Costs / Risks

* Adds control-plane components (Istio, Flagger)
* Requires careful tuning to avoid noisy rollbacks
* Resource overhead (sidecars) may be non-trivial
* Debugging becomes more complex due to additional network layer

### Mitigations

* Start with **phased rollout**:

  * enable Istio in a limited set of namespaces
  * deploy canaries for selected workloads first

* Establish conservative defaults:

  * canary step weight and intervals
  * error-rate thresholds
  * latency thresholds

* Require minimum observability readiness:

  * metrics must be available and stable before enabling automation

* Provide break-glass controls:

  * ability to pause Flagger
  * ability to disable sidecar injection on a namespace

## Security & Compliance Implications

* **mTLS** strengthens confidentiality and integrity for east/west traffic.

* **Policy enforcement**:

  * mesh policies and Kubernetes network policies should align (no conflicting rules)

* **Auditability**:

  * canary policies and thresholds are version-controlled
  * promotions/rollbacks are visible in Git history and cluster events

* **Evidence generation**:

  * canary events and analysis outcomes can be retained as artifacts
  * dashboards and alerts provide operational evidence

## Implementation Notes

### Installation approach

* Deploy Istio via Helm or operator in a GitOps-managed manner.
* Deploy Flagger via Helm in GitOps-managed manner.

### Namespace adoption strategy

* Use explicit namespace labels for sidecar injection:

  * `istio-injection=enabled`

* Default is opt-in.

### Canary definition pattern

* Each canary is declared with:

  * target Deployment
  * analysis interval
  * step weight progression
  * max failed checks
  * metrics thresholds (e.g., error rate, latency)

* Canary definitions live with the workload manifests in the repo.

### Standard signals (baseline)

At minimum, canary analysis should consider:

* HTTP 5xx error rate
* request duration (p95 or p99)
* success rate
* optionally: custom business KPIs (app-specific)

Data sources:

* Prometheus/Mimir for metrics
* Loki for logs (optional for canary decisions)
* Tempo for traces (primarily for debugging, optional for automated gating)

### Operational controls

* Provide standardized “pause” and “resume” patterns:

  * suspend Flagger canary
  * rollback via Git revert

* Define runbooks:

  * how to interpret canary failures
  * how to override thresholds temporarily (with approvals)

## Validation

This ADR is successful if:

* Canary deployments can be performed in Development and Staging reliably
* Canary analysis promotes healthy releases automatically
* Regressions cause automatic rollback with clear observability evidence
* Production rollouts are safer and measurable

Suggested test cases:

* deploy a canary with an intentional error to confirm rollback
* deploy a healthy canary and confirm automated promotion
* simulate telemetry loss and confirm Flagger fails safe

## Rollback Plan

If mesh + canary automation causes instability:

* disable Flagger canary resources (Git revert)
* pause or remove Flagger controller
* disable sidecar injection for affected namespaces

If long-term mesh complexity is unacceptable:

* create a superseding ADR and migrate to ingress-only canaries or a different mesh

## Follow-ups

* Define default canary templates and threshold profiles per environment
* Decide whether to enforce mTLS strict mode in all namespaces or phase it
* Add policy-as-code rules to require:

  * readiness probes
  * resource requests/limits
  * standardized labels/annotations
  * safe rollout strategy markers
