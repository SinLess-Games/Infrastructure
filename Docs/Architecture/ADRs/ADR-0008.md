## Context

The platform requires **progressive delivery** for Kubernetes workloads.

A standard Kubernetes rolling update is useful, but it does not provide enough
control for production-grade release safety. The platform needs a rollout model
that can gradually expose users to new application versions, evaluate telemetry,
and automatically stop or roll back bad releases.

The platform needs:

- gradual traffic shifting
- automated health analysis
- fast rollback on regression
- repeatable rollout patterns
- observable rollout state
- clear promotion gates
- Git-based auditability
- support for production and non-production environments

The broader platform promotion model is GitOps-based.

Application and platform changes move through:

```text
dev → staging → prod
```

Promotion between environments happens through pull requests. Inside a specific
cluster, however, the workload still needs a safe rollout strategy.

The platform also has a zero-trust posture. East/west service traffic should be
secured with service identity, policy, and mutual TLS where appropriate. A
service mesh provides the traffic-management primitives needed for canaries and
also supports the zero-trust networking model.

Observability is required for progressive delivery. Automated rollout decisions
must be based on telemetry from the running system.

At minimum, canary analysis requires:

- request success signals
- HTTP error counts
- latency signals
- workload readiness
- deployment health
- rollback visibility

---

## Decision

Adopt **Istio** as the service mesh and **Argo Rollouts** as the progressive
delivery controller for in-cluster canary and blue/green deployments.

Argo CD remains the GitOps controller.

The responsibility split is:

| Component | Responsibility |
| --- | --- |
| Argo CD | Sync declared manifests from Git |
| Argo Rollouts | Control rollout progression and promotion |
| Istio | Shift live traffic between stable and canary versions |
| Prometheus/Mimir | Provide rollout analysis metrics |
| GitHub pull requests | Promote changes between environments |

Argo CD should not directly manage live canary traffic weights during rollout.
Argo CD applies the desired resources from Git. Argo Rollouts then controls the
active rollout by updating Istio routing weights and Kubernetes service
selectors as the rollout progresses.

This means Argo CD must be configured to ignore the live fields that Argo
Rollouts is expected to mutate.

Those fields include:

- stable Service selectors
- canary Service selectors
- Istio VirtualService route weights

---

## Scope

This ADR governs:

- the standard progressive delivery controller
- the use of Istio for traffic shifting
- the use of Argo Rollouts for canary and blue/green deployment control
- the integration pattern between Argo CD, Argo Rollouts, Istio, and metrics
- the expectation that rollout definitions are stored in Git
- the high-level rollout safety model for Kubernetes workloads

This ADR does not:

- define every rollout threshold
- define every application-specific metric
- define final production SLOs
- define the full ingress architecture
- define complete Argo CD Application manifests
- define the full Istio security model
- replace environment promotion policy

Environment promotion remains Git-based:

```text
dev → staging → prod
```

This ADR specifically covers **in-cluster progressive delivery** after a change
has been synced into a cluster.

## Standard Progressive Delivery Model

The standard model is:

```text
GitHub repository
  ↓
Argo CD syncs manifests
  ↓
Argo Rollouts creates or updates ReplicaSets
  ↓
Istio shifts traffic between stable and canary services
  ↓
Prometheus/Mimir provides analysis metrics
  ↓
Argo Rollouts promotes, pauses, or aborts the release
```

For normal HTTP services, the preferred starting pattern is **host-level traffic
splitting**.

Each canary-enabled workload should generally have:

- one `Rollout`
- one stable `Service`
- one canary `Service`
- one Istio `VirtualService`
- one or more `AnalysisTemplate` resources
- one Argo CD `Application` or parent ApplicationSet entry

Example resource set:

```text
rollout.yaml
service-stable.yaml
service-canary.yaml
virtualservice.yaml
analysis-template.yaml
kustomization.yaml
```

---

## Recommended Traffic Splitting Pattern

Use Istio host-level traffic splitting first.

In this model, the Istio `VirtualService` routes traffic between two Kubernetes
services:

- `app-stable`
- `app-canary`

Argo Rollouts updates the service selectors and VirtualService weights as the
canary progresses.

Example routing state at the beginning of a rollout:

```text
stable: 100%
canary:   0%
```

Example routing state during a rollout:

```text
stable: 75%
canary: 25%
```

Example routing state after promotion:

```text
stable: 100%
canary:   0%
```

The canary service is a control mechanism. After promotion, the promoted version
becomes the stable version.

## Alternatives Considered

### A1) No Progressive Delivery

Use only default Kubernetes rolling updates.

**Pros:**

- simplest option
- no additional rollout controller
- fewer resources to manage

**Cons:**

- limited traffic control
- weak automated rollback behavior
- harder to perform metric-based release gates
- less visibility into release progression
- less useful for production release safety

This option is rejected for production workloads because it does not provide
enough control or telemetry-based gating.

---

### A2) Flagger with Istio

**Pros:**

- mature canary automation
- strong metric-based rollout support
- works well with Istio
- good fit for FluxCD-based platforms

**Cons:**

- less aligned with the chosen Argo CD GitOps model
- introduces a second progressive delivery model outside the Argo ecosystem
- would make ADR-0007 and ADR-0008 less consistent
- Argo Rollouts better matches the selected Argo CD platform direction

Flagger is rejected because the platform standard is Argo CD and Argo Rollouts.

---

### A3) Argo Rollouts without Istio

Argo Rollouts can be used without Istio by relying only on Kubernetes service
switching or blue/green style promotions.

**Pros:**

- simpler than using a full service mesh
- fewer mesh dependencies
- still provides rollout objects and promotion controls

**Cons:**

- weaker traffic shaping for percentage-based canaries
- less useful for internal service-to-service rollout control
- does not address zero-trust east/west traffic requirements
- does not provide the same service-mesh policy foundation

This option may be acceptable for simple internal workloads, but it is not the
platform standard.

---

### A4) Ingress Controller Canary Features

Examples:

- NGINX canary annotations
- Traefik weighted services
- gateway-level manual traffic splits

**Pros:**

- less control-plane overhead
- useful for ingress-only applications
- easier initial setup

**Cons:**

- primarily controls north/south ingress traffic
- does not solve east/west progressive delivery
- does not provide a complete zero-trust service mesh model
- can lead to inconsistent rollout strategies across workloads

This option is rejected as the primary platform pattern.

---

### A5) Linkerd with Flagger or SMI

**Pros:**

- lighter service mesh
- simpler operational profile
- strong mTLS defaults

**Cons:**

- smaller traffic-management feature set than Istio
- different operational model than the selected Istio direction
- less aligned with desired gateway and mesh policy flexibility

This option is rejected because Istio is the chosen service mesh direction.

## Rationale

### 1. Alignment with Argo CD

Argo Rollouts fits naturally with Argo CD.

Argo CD handles desired state synchronization. Argo Rollouts handles release
progression. This keeps GitOps and rollout automation separated cleanly.

Argo CD should answer:

- what should exist?
- what is synced?
- what is out of sync?
- what is healthy?

Argo Rollouts should answer:

- what version is stable?
- what version is canary?
- what traffic weight is active?
- should the rollout continue?
- should the rollout pause?
- should the rollout abort?

---

### 2. Safer Production Releases

Canary rollouts reduce blast radius.

Instead of sending all production traffic to a new version immediately, traffic
can move through controlled steps.

Example production pattern:

```text
5% → pause → analyze
25% → pause → analyze
50% → pause → analyze
100% → promote
```

Bad releases should be detected before they affect all users.

---

### 3. Istio Provides Traffic Control

Istio provides the traffic-management layer needed for controlled routing.

The platform can use Istio for:

- weighted routing
- service-to-service policy
- mTLS
- request telemetry
- gateway integration
- future advanced routing patterns

Progressive delivery and zero-trust networking benefit from the same mesh layer.

---

### 4. Metrics-Driven Decisions

Argo Rollouts can use analysis templates to query Prometheus-compatible metrics.

This allows rollout decisions to be based on observed behavior rather than
manual inspection alone.

Baseline signals should include:

- HTTP 5xx count
- HTTP 4xx count where relevant
- request success rate
- p95 or p99 latency
- pod readiness
- application-specific business metrics where appropriate

---

### 5. Git-Based Auditability

Rollout definitions live in Git.

This means release behavior is:

- reviewable
- versioned
- reproducible
- auditable
- consistent across environments

The canary strategy itself becomes part of the declared platform state.

---

## Consequences

### Positive

- safer production deployments
- reduced blast radius
- automated rollback options
- better release visibility
- stronger integration with Argo CD
- consistent rollout patterns across environments
- clear separation between GitOps sync and rollout control
- stronger support for zero-trust traffic management through Istio
- better operational evidence from rollout history and metrics

---

### Costs and Risks

- adds Argo Rollouts as a critical platform controller
- requires Istio traffic resources to be correct
- requires reliable metrics before automation can be trusted
- Argo CD can fight Argo Rollouts if ignore rules are wrong
- rollout debugging is more complex than plain Deployments
- analysis thresholds can be too strict or too loose
- sidecar overhead must be considered for mesh-enabled workloads

---

### Mitigations

- start with a limited set of workloads
- require readiness and liveness probes
- require resource requests and limits
- validate manifests in CI
- use conservative production rollout steps
- use clear Argo CD ignore differences for Rollouts-managed fields
- alert on degraded Rollouts and failed analyses
- keep emergency abort and promote commands documented
- require manual production promotion for high-risk workloads
- tune thresholds using real baseline metrics before strict enforcement

---

## Security and Compliance Implications

### mTLS and Service Identity

Istio supports the platform zero-trust model by enabling service identity and
mutual TLS for service-to-service communication.

Progressive delivery should not bypass mesh security controls.

---

### Change Control

Production rollout definitions are stored in Git and reviewed through pull
requests.

Production-impacting rollout changes should require:

- pull request review
- CI validation
- manifest rendering checks
- policy validation
- clear rollback notes where appropriate

---

### Audit Evidence

Progressive delivery creates multiple evidence points:

- pull request history
- commit history
- Argo CD sync history
- Argo Rollouts rollout history
- AnalysisRun results
- Kubernetes events
- Prometheus/Mimir metrics
- Grafana dashboards
- alert history

These artifacts support incident review, compliance review, and operational
postmortems.

---

### Secret Handling

Rollout manifests must not contain plaintext secrets.

Approved secret patterns include:

- External Secrets Operator with Vault
- Vault CSI where appropriate
- references to existing Kubernetes secrets created by approved secret workflows

---

### Policy Enforcement

Policy-as-code should eventually enforce progressive delivery safety standards.

Examples:

- require probes on canary workloads
- require resource requests and limits
- require standard labels
- require rollout ownership metadata
- restrict production auto-sync
- require approved traffic-routing patterns
- prevent plaintext secrets in rollout manifests

## Implementation Notes

### Controller Installation

Install and manage the following components through GitOps:

- Argo CD
- Argo Rollouts
- Istio control plane
- Prometheus-compatible metrics backend
- Grafana dashboards and alerts

Argo Rollouts should be installed before workloads that use the `Rollout` kind.

---

### Namespace Adoption Strategy

Mesh adoption should be explicit.

Recommended default:

```yaml
metadata:
  labels:
    istio-injection: enabled
```

Use namespace-level opt-in for workloads that are ready for sidecar injection.

Do not enable mesh injection globally until the platform has validated:

- DNS behavior
- sidecar resource overhead
- service compatibility
- mTLS policy behavior
- observability coverage
- rollout behavior

---

### Standard Rollout Shape

A canary-enabled application should generally use this resource shape:

```text
Rollout
Service stable
Service canary
VirtualService
AnalysisTemplate
```

The stable and canary services must point at the same logical application but are
managed by Argo Rollouts during the release.

---

### Example Rollout Strategy

A reasonable starting strategy for production HTTP workloads:

```yaml
strategy:
  canary:
    stableService: example-stable
    canaryService: example-canary
    trafficRouting:
      istio:
        virtualService:
          name: example
          routes:
            - primary
    steps:
      - setWeight: 5
      - pause:
          duration: 5m
      - analysis:
          templates:
            - templateName: example-success-check
      - setWeight: 25
      - pause:
          duration: 10m
      - analysis:
          templates:
            - templateName: example-success-check
      - setWeight: 50
      - pause:
          duration: 10m
      - analysis:
          templates:
            - templateName: example-success-check
      - setWeight: 100
```

This should be adjusted per workload and environment.

---

### Environment Defaults

Recommended starting defaults:

| Environment | Sync Policy | Canary Behavior | Analysis Strictness |
| --- | --- | --- | --- |
| dev | automated | fast steps, short pauses | lenient |
| staging | automated | production-like steps | moderate |
| prod | manual or approved automated | slow steps, longer pauses | strict |

Production canaries should prioritize stability over speed.

---

### Argo CD Ignore Differences

Argo CD must ignore fields that Argo Rollouts mutates during active canaries.

At minimum, applications using Istio canaries should ignore:

- stable Service selector changes
- canary Service selector changes
- VirtualService route weight changes

Example Argo CD ignore differences:

```yaml
ignoreDifferences:
  - group: ""
    kind: Service
    name: example-stable
    namespace: example
    jsonPointers:
      - /spec/selector

  - group: ""
    kind: Service
    name: example-canary
    namespace: example
    jsonPointers:
      - /spec/selector

  - group: networking.istio.io
    kind: VirtualService
    name: example
    namespace: example
    jqPathExpressions:
      - .spec.http[]?.route[]?.weight
```

The Argo CD Application should also use:

```yaml
syncOptions:
  - RespectIgnoreDifferences=true
```

Without these settings, Argo CD may attempt to reset the fields that Argo
Rollouts is intentionally changing.

---

### Standard Analysis Signals

Baseline analysis should start with Prometheus-compatible metrics.

Minimum recommended signals:

- HTTP 5xx count
- request success rate
- p95 latency
- pod readiness
- restart count

Optional advanced signals:

- HTTP 4xx count by route
- saturation metrics
- queue depth
- custom business metrics
- trace-derived latency
- log-derived error patterns

For production, avoid promoting only on pod readiness. Readiness means the pod
can receive traffic. It does not prove the new version behaves correctly.

---

### Rollout Controls

Operators should have standard commands for emergency control.

Watch a rollout:

```bash
kubectl argo rollouts get rollout <name> -n <namespace> --watch
```

Promote a paused rollout:

```bash
kubectl argo rollouts promote <name> -n <namespace>
```

Abort a bad rollout:

```bash
kubectl argo rollouts abort <name> -n <namespace>
```

Retry a rollout after fixing the issue:

```bash
kubectl argo rollouts retry rollout <name> -n <namespace>
```

Undo to the previous revision:

```bash
kubectl argo rollouts undo <name> -n <namespace>
```

## Validation

This ADR is successful when:

- Argo Rollouts is installed and healthy
- Istio is installed and healthy
- Argo CD can sync Rollout resources
- canary traffic shifts through Istio VirtualServices
- Argo CD does not fight Rollouts-managed live fields
- healthy canaries promote successfully
- bad canaries abort or roll back automatically
- rollout state is visible through Argo Rollouts and Grafana
- production rollout evidence can be collected after a deployment

Suggested validation tests:

- deploy a healthy canary and confirm promotion
- deploy a canary that returns intentional 5xx responses and confirm abort
- manually change a VirtualService weight and confirm ownership behavior
- confirm Argo CD ignore differences prevent sync fighting
- confirm AnalysisRun results are visible
- confirm rollout alerts fire on degraded state
- confirm mTLS does not break the canary path
- confirm rollback restores the previous stable version

---

## Rollback Plan

If a specific canary release fails:

1. abort the rollout
2. confirm traffic returns to the stable ReplicaSet
3. inspect AnalysisRun results
4. inspect application logs and metrics
5. fix the issue in Git
6. sync the corrected change through Argo CD
7. retry the rollout

If Argo Rollouts causes platform instability:

1. pause affected rollouts
2. disable automated promotion where needed
3. revert the Git change that introduced the issue
4. sync the known-good manifests
5. validate stable service routing
6. investigate the Argo Rollouts controller

If Istio traffic routing causes instability:

1. stop or abort the active rollout
2. set traffic back to the stable service
3. inspect the VirtualService and DestinationRule resources
4. inspect Envoy sidecar configuration
5. validate gateway and service routing
6. revert the related Git change if needed

If long-term complexity becomes unacceptable:

- create a superseding ADR
- define an alternate progressive delivery model
- migrate selected workloads back to rolling updates or another rollout strategy
- document the migration and validation plan

---

## Operational Guardrails

Production workloads should not use progressive delivery until they have:

- readiness probes
- liveness probes where appropriate
- resource requests and limits
- stable service metrics
- dashboards
- rollback instructions
- defined ownership
- clear alert routing
- tested canary failure behavior

Production canaries should not rely only on manual observation.
