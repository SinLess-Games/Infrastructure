## Context

The platform is built to a **Zero Trust** and **compliance-ready** baseline. That baseline requires:

* Clear trust boundaries between management, storage, and workloads
* Reduced blast radius (a compromise in one plane should not automatically expose others)
* Observable and enforceable network policy
* Support for HA services (VIPs, LBs) and multi-domain ingress
* A topology that can expand (more nodes, additional clusters, future hybrid burst)

The infrastructure includes:

* Proxmox cluster with multiple nodes (`pve-01` … `pve-05`)
* Multiple NIC capabilities, including SFP+ 10Gb on some nodes
* A storage backend (Ceph) that benefits from a dedicated replication network
* Kubernetes clusters requiring dedicated node/control-plane traffic separation
* A DMZ exposure model that must be **Cloudflare-first** (tunnel-based, not inbound port exposure)

The network design must therefore separate:

* Management and automation access
* Platform services (Vault, Authentik, MinIO, GitLab, DNS)
* Kubernetes node and control-plane traffic
* Storage replication and backplane traffic
* DMZ/ingress traffic surfaces

## Decision

Standardize the entire platform on **five VLAN-backed network planes**, with **default-deny** posture between planes and **explicit allow lists**.

### VLAN planes

* **VLAN 10 — Management**

  * Proxmox UI/API/SSH
  * automation controllers and jump hosts
  * iDRAC/management where applicable (if present, may be separated later into an OOB VLAN)
  * **Access policy:** WireGuard overlay and/or Cloudflare Access only

* **VLAN 20 — Infra/Services**

  * Vault, Authentik, MinIO, GitLab, Technitium DNS
  * shared internal APIs and service endpoints
  * **Access policy:** restricted; no public inbound

* **VLAN 30 — Kubernetes/Nodes**

  * Kubernetes node traffic (kubelet, control-plane)
  * internal load balancers / service VIPs
  * cluster east/west services not intended for public exposure
  * **Access policy:** restricted; no public inbound

* **VLAN 40 — Storage**

  * Ceph replication (cluster network)
  * storage backplane traffic
  * **Access policy:** strictly restricted to storage participants

* **VLAN 50 — DMZ**

  * ingress landing surfaces (reverse proxies, gateways as needed)
  * **Primary exposure path:** Cloudflare Tunnel
  * **Access policy:** no direct inbound to management; any exposure is tightly scoped

### Routing and boundary enforcement

* Inter-VLAN routing is performed only where required and is policy-gated.
* The default stance is **deny cross-plane**.
* Allowed flows are documented as “contracts” (source plane → destination plane + ports + purpose).

### Host-level enforcement

* Proxmox host firewall is enabled and managed as code.
* Kubernetes NetworkPolicies (and mesh mTLS where used) enforce workload-level boundaries.

## Scope

This ADR governs:

* Which network planes exist and what they are used for
* The requirement that planes are isolated by default
* The intent that Ceph replication uses VLAN 40
* The intent that management access is never public

This ADR does not:

* Specify exact IP ranges (these are environment-defined)
* Specify the final CNI choice (Cilium vs Calico decision occurs in a separate ADR)
* Replace detailed switch configuration runbooks (those belong under `Docs/Network/`)

## Alternatives Considered

### A1) Flat network (single VLAN)

**Pros**

* Simplest to deploy and troubleshoot

**Cons**

* Large blast radius
* Weak compliance story (segmentation controls are difficult to evidence)
* Harder to enforce least privilege
* Storage replication and management traffic mix with workloads

### A2) SDN overlays only (no VLAN segmentation)

**Pros**

* Flexible
* Can provide strong isolation logically

**Cons**

* Higher complexity and operational overhead early
* Troubleshooting is harder without a simple physical/logical baseline
* Requires mature tooling and strong standards from day one

### A3) Full VRF-driven enterprise segmentation

**Pros**

* Very strong isolation
* Clear routing policy boundaries

**Cons**

* Increased complexity and equipment/config burden
* Not necessary as an initial baseline for this platform

### A4) “VPN for everything” with minimal internal segmentation

**Pros**

* Easy access control story

**Cons**

* VPN does not replace segmentation
* A compromise inside the VPN still has broad lateral movement options

## Rationale

This VLAN plane model is selected because it:

1. **Supports Zero Trust and compliance controls**

   * Segmentation is explicit and reviewable.
   * Default-deny boundaries can be audited and evidenced.

2. **Improves performance and reliability**

   * Ceph replication benefits from dedicated VLAN 40.
   * Kubernetes control-plane/node traffic is separated from DMZ and storage.

3. **Simplifies day-2 operations**

   * Clear troubleshooting boundaries (“what plane is this traffic on?”).
   * Predictable policy design and capacity planning.

4. **Scales to future expansion**

   * Adding nodes or clusters is a matter of attaching them to the correct planes.
   * Hybrid burst and additional sites can map to the same plane model.

## Consequences

### Positive

* Reduced blast radius and lateral movement paths
* Easier policy-as-code enforcement and auditability
* Better storage and workload performance isolation

### Costs / Risks

* Requires consistent switch configuration and tagging
* Requires disciplined IPAM and naming conventions
* Mis-tagging a port or VM bridge can create subtle failures

### Mitigations

* Maintain switch config templates and diagrams in-repo

* Provide validation tasks:

  * VLAN tagging checks
  * reachability tests per plane
  * firewall policy verification

* Enforce host firewall rules as code

## Security & Compliance Implications

* **Least privilege network access** is enforceable and auditable.

* **Management plane isolation** reduces the likelihood of remote compromise.

* **Storage plane isolation** helps protect confidentiality and availability of data.

* **Evidence generation** is supported by:

  * firewall configuration as code
  * change history via PRs
  * periodic validation runs producing artifacts

## Implementation Notes

### Physical and link-layer intent

* Nodes with 2-port SFP+ NICs connect both ports to the aggregation switch.
* Where supported, bonds should use LACP (802.3ad) to provide redundancy and throughput.
* VLANs are trunked over the bonded uplinks; per-plane bridges attach VLAN subinterfaces.

### Proxmox bridge conventions (recommended)

The exact naming may evolve, but the intent is:

* `vmbr0` → VLAN 10 (Management)
* `vmbr1` → VLAN 20 (Infra/Services)
* `vmbr2` → VLAN 30 (Kubernetes/Nodes)
* `vmbr3` → VLAN 40 (Storage)
* `vmbr4` → VLAN 50 (DMZ)

If a single physical bond carries all VLANs, bridges may use VLAN-aware configuration.

### Firewall defaults

* Proxmox firewall enabled at:

  * Datacenter level (global baseline)
  * Node level (node-specific constraints)
  * VM level (where appropriate)

* Default policy:

  * **DROP inbound** unless explicitly allowed
  * Allow only necessary east/west flows between plane endpoints

### Plane “contracts” (examples)

These are conceptual examples; the authoritative list lives in code.

* VLAN 10 → VLAN 20: automation to Vault/Authentik/GitLab admin endpoints (restricted ports)
* VLAN 30 → VLAN 40: Kubernetes storage clients to Ceph public/client network (if exposed)
* VLAN 50 → VLAN 30: ingress gateway to Kubernetes services (only needed ports)
* VLAN 10 → VLAN 30: kubectl/admin tooling to API endpoints (restricted)

### Cloudflare-first DMZ

* Public exposure is handled via Cloudflare Tunnel.
* DMZ services should not expose inbound ports directly to the Internet.

### Documentation and source-of-truth

* IP ranges, host IPs, and DNS names are defined per environment.
* Network diagrams and port maps are maintained in `Docs/Network/`.

## Validation

This ADR is successful if:

* VLANs are consistently present end-to-end (switch → host → bridge → VM)
* Proxmox management services are reachable only via approved access paths
* Ceph replication traffic is isolated to VLAN 40
* Kubernetes nodes communicate correctly on VLAN 30
* DMZ traffic surfaces do not expose management planes
* Automated tests confirm default-deny boundaries (only approved flows succeed)

## Rollback Plan

If segmentation causes unacceptable operational friction during initial bring-up:

* Temporarily reduce to a minimal subset:

  * VLAN 10 (Management)
  * VLAN 30 (Kubernetes/Nodes)
  * VLAN 40 (Storage)

* Keep naming and plane intent intact.

* Reintroduce VLAN 20 and VLAN 50 once baseline stability is achieved.

If a major redesign is required:

* Create a new ADR to **supersede** this one.
* Provide a migration plan with:

  * switch port changes
  * host bridge changes
  * VM NIC reattachment steps

## Follow-ups

* Create detailed switch port maps (per node and per NIC)
* Define the authoritative allow-list matrix (plane-to-plane port contracts)
* Decide on Kubernetes CNI and whether to encrypt pod traffic at L3
* Add validation tasks and CI artifacts for network policy verification
