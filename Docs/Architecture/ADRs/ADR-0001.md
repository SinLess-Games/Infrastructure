## Context

The SinLess Games infrastructure platform spans multiple tightly-coupled layers that must evolve together:

* **Bare metal bootstrap and node onboarding** (PXE/netboot, first-boot bootstrap)
* **Proxmox VE** installation and configuration (cluster formation, firewalling, storage, RBAC)
* **Storage** (Proxmox-integrated Ceph pools/OSD groups, VM storage policies)
* **VM image lifecycle** (Packer templates and template promotion)
* **VM provisioning** (Terraform modules, environment plans, state, drift detection)
* **Kubernetes clusters** (Rancher K3s-family distribution via Rancher ecosystem tooling)
* **GitOps** cluster state management (FluxCD) and promotions (dev → staging → prod)
* **Security & Zero Trust** (Vault, WireGuard, OIDC via Authentik, Cloudflare Tunnel/Access)
* **Observability** (Grafana stack, OnCall, InfluxDB for Proxmox metrics)
* **Backups and DR** (Proxmox Backup Server, Velero to MinIO)
* **Compliance controls** (policy-as-code, audit evidence, change control)

These components have strong operational coupling:

* Networking/VLAN changes affect Proxmox bridges, Ceph replication, Kubernetes node networks, and service endpoints.
* Certificate strategy changes affect Cloudflare, cert-manager, Vault, and ingress/tunnels.
* Backup strategy changes affect PBS schedules, Velero storage and credentials, MinIO, retention, and restore runbooks.

If configuration is fragmented across multiple repositories, the platform is prone to:

* **Version skew** (Terraform module version doesn’t match Ansible roles or cluster manifests)
* **Documentation drift** (docs no longer reflect deployed reality)
* **Promotion complexity** (multiple repos must be promoted in lockstep)
* **Reduced auditability** (harder to show who changed what, when, and why)

The platform also explicitly requires:

* **Auditable change control** (PRs, approvals, immutable history)
* **Repeatability** (ability to rebuild from scratch)
* **GitOps promotion** (environment overlays and reconciled state)
* **Evidence generation** (plans, scans, policy checks as artifacts)

## Decision

Adopt a **single monorepo** as the canonical “source of truth” for:

* IaC (Terraform)
* Host configuration (Ansible)
* VM images (Packer)
* Kubernetes platform + apps (FluxCD GitOps)
* PXE/netboot assets (provisioning configs/scripts)
* Security & compliance controls (policy-as-code, scans, evidence outputs)
* Documentation (architecture, ADRs, operational runbooks)

Environment variance is expressed through:

* `Environments/Development/`
* `Environments/Staging/`
* `Environments/Production/`

Orchestration is standardized via:

* **go-task** (`Taskfile.yaml` and domain Taskfiles)

## Scope

This ADR governs:

* Repository structure and ownership
* Where the “intended state” is defined and maintained
* How changes are proposed, reviewed, approved, and promoted between environments

This ADR does **not** mandate a single runtime or platform product. It mandates a single, authoritative **configuration and change-control system**.

## Alternatives Considered

### A1) Multi-repo per domain (Terraform repo, Ansible repo, Kubernetes repo, etc.)

**Pros**

* Smaller repos; simpler per-repo tooling
* Clear boundaries per domain

**Cons**

* Cross-domain changes require synchronized PRs across multiple repos
* Higher drift risk and coordination overhead
* Promotions and releases become “multi-repo choreography”
* Harder onboarding: multiple repos, multiple docs, multiple conventions

### A2) Monorepo + heavy use of git submodules

**Pros**

* Easy to pin upstream projects (e.g., installers/operators) at known commits

**Cons**

* Submodule operational complexity (init/update pitfalls)
* Tooling friction with nested repos
* Higher failure rate during onboarding and CI

**Notes**

* Submodules may be used **selectively** for vendoring specific upstream content, but should be the exception, not the norm.

### A3) Docs separate from code

**Pros**

* Docs can move independently and quickly

**Cons**

* Documentation drift becomes inevitable
* Operational and compliance trust is reduced (“docs are aspirational, not authoritative”)

## Rationale

A monorepo best supports the platform goals:

1. **Atomic changesets**

   * One PR can include networking changes, Proxmox automation updates, and documentation updates.

2. **Promotion through Git**

   * Development → Staging → Production is expressed as code changes and PR promotion, consistent with GitOps.

3. **Auditability & evidence**

   * PR history + CI artifacts become durable evidence of change control and review.

4. **Reduced drift**

   * Flux reconciles Kubernetes from the repo; Terraform plans expose infra drift; Ansible enforces host state.

5. **Operational clarity**

   * New operators clone one repo to understand and run the platform.

## Consequences

### Positive

* Single authoritative location for intended state
* Consistent conventions and reusable modules across domains
* Centralized dependency automation (Renovate/Mend configuration in one place)

### Costs / Risks

* CI becomes more complex (multi-toolchain validation)
* Repository sprawl risk if conventions are not enforced
* Strong standards needed for:

  * directory structure
  * naming
  * environment overlays
  * secrets handling
  * doc updates and ADR hygiene

### Mitigations

* Enforce conventions via:

  * Taskfile entrypoints (`task lint`, `task validate`, `task env:plan`, `task env:apply`)
  * CI lint/validate gates on PR
  * CODEOWNERS + protected branches + required reviews
  * “Material changes require ADR reference” rule

## Security & Compliance Implications

* **Change control**

  * Production changes must go through PR review + approvals.
  * Protected branches enforce minimum reviewers and passing checks.

* **Evidence generation**

  * CI artifacts include:

    * Terraform plans
    * validation outputs (lint, policy checks)
    * security scan results
    * promotion PR records

* **Secrets hygiene**

  * No secrets committed to Git.
  * Vault is the system of record for secrets.
  * CI and automation retrieve secrets using approved auth methods and least privilege.

* **Least privilege**

  * Separate credentials per environment and per pipeline scope.
  * Short-lived credentials where feasible (OIDC/JWT-based to Vault).

## Implementation Notes

### Minimum repository layout

* `Ansible/` — Proxmox and host configuration
* `Terraform/` — Proxmox VM provisioning, shared modules
* `Packer/` — image templates
* `Kubernetes/` — FluxCD-managed manifests (clusters/base/apps)
* `PXE/` — PXE services and unattended install assets
* `Policy/` — policy-as-code checks
* `Docs/` — architecture + operations + ADRs
* `Environments/` — env overlays and variables

### Workflow conventions

* PRs run:

  * format/lint/validate for all domains
  * plans (at minimum Terraform plan; optionally “diff” for cluster manifests)
* Applies run:

  * on merge
  * gated by environment approvals for staging/prod

### Documentation rules

* Material changes require:

  * updates to `Docs/Architecture/ARCHITECTURE.md`, and/or
  * a new ADR under `Docs/Architecture/ADRs/`

## Validation

This ADR is successful if:

* A full environment rebuild is possible using only the repo + documented secrets bootstrap procedure
* CI reliably blocks invalid Terraform/Ansible/Kubernetes changes
* Promotions dev → staging → prod are PR-based with clear diffs and approvals
* Drift is minimized:

  * Flux reconciles Kubernetes
  * Terraform plan is stable (no unexpected changes) when environment is healthy

## Rollback Plan

This decision is foundational and typically not rolled back as a single change.

If future complexity demands splitting the repo:

* Extract domain directories into separate repos **with preserved version pinning**
* Maintain an “umbrella” repo for environment promotion and shared standards
* Create a new ADR that **supersedes** this one and documents the transition plan

## Follow-ups
