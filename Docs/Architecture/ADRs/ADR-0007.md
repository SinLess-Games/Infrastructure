## Context

The Infrastructure platform is designed to be operated through **Git-driven change management**.

The platform needs:

- reproducible cluster state
- minimal configuration drift
- auditable change control
- clear environment separation
- repeatable cluster recovery
- promotion through controlled pull requests
- visibility into synchronization and health status
- a clean operational model for Kubernetes platform components and workloads

Kubernetes resources, platform services, and application definitions must be managed declaratively.

This includes, but is not limited to:

- namespaces
- RBAC
- NetworkPolicies
- Istio resources
- cert-manager resources
- ExternalDNS resources
- External Secrets resources
- observability components
- security tooling
- application workloads
- Helm chart releases
- Kustomize overlays

A GitOps controller is required to continuously reconcile the desired state in Git with the actual state running in each Kubernetes cluster.

---

## Decision

Adopt **Argo CD** as the GitOps controller for all Kubernetes clusters.

Argo CD is responsible for:

- continuously reconciling desired cluster state from the Infrastructure monorepo
- detecting and reporting configuration drift
- managing Kubernetes manifests, Kustomize overlays, and Helm-rendered resources
- providing visibility into application sync status and health
- enabling controlled promotion through Git pull requests
- supporting app-of-apps and ApplicationSet patterns for scalable cluster management

Each cluster is bootstrapped from a dedicated Git path.

Recommended cluster paths:

- `Kubernetes/clusters/dev/`
- `Kubernetes/clusters/staging/`
- `Kubernetes/clusters/prod/`

Platform components and applications are organized as:

- shared bases
- environment overlays
- namespace-scoped applications
- cluster-specific bootstrap manifests

Promotion model:

- **dev → staging → prod**
- promotion is performed by pull requests
- pull requests update versions, manifests, overlays, or Helm values
- CI validates the requested change before Argo CD reconciles it

---

## Scope

This ADR governs:

- the selection of **Argo CD** as the GitOps controller
- the repository-to-cluster reconciliation model
- the high-level cluster bootstrap structure
- the principle that promotion happens through Git
- the expectation that Argo CD is the source of Kubernetes reconciliation truth

This ADR does not:

- define the complete repository layout
- define every Application or ApplicationSet
- define CI pipelines
- define every progressive delivery rule
- define every policy-as-code control
- define the full secrets management implementation

Those details are covered by implementation documentation and related ADRs.

## Alternatives Considered

### A1) FluxCD

**Pros:**

- strong Kubernetes-native GitOps model
- good Kustomize and Helm support
- strong reconciliation behavior
- lightweight controller design
- mature image automation options

**Cons:**

- weaker built-in visual interface compared to Argo CD
- less operator-friendly for manual inspection
- less aligned with the desired dashboard-driven GitOps workflow
- progressive delivery would likely lean toward Flagger instead of Argo Rollouts

FluxCD remains a strong option, but Argo CD better fits this platform’s desired operational visibility and GitOps user experience.

---

### A2) CI-only Deployments

Examples:

- `kubectl apply` from GitHub Actions
- `helm upgrade` from GitHub Actions
- direct deployment scripts from CI

**Pros:**

- simple initial setup
- fewer in-cluster components
- easy to understand for small environments

**Cons:**

- not true continuous reconciliation
- drift can occur silently
- cluster state depends on the last successful CI run
- rollback is less clean
- operational visibility is weaker
- manual repair often becomes necessary

CI-only deployment is rejected because the platform requires continuous reconciliation and drift detection.

---

### A3) Manual Operations

Examples:

- direct `kubectl apply`
- manual Helm installs
- ad hoc operator changes

**Pros:**

- immediate control
- useful for emergency investigation
- low initial tooling overhead

**Cons:**

- high drift risk
- poor auditability
- inconsistent environments
- difficult disaster recovery
- difficult promotion tracking
- unsuitable for production operations

Manual operations are only acceptable as a documented break-glass procedure.

## Rationale

Argo CD is selected because it provides the best fit for the platform’s desired operational model.

### 1. Strong GitOps Visibility

Argo CD provides a clear UI and API for understanding:

- what is deployed
- what is out of sync
- what is healthy
- what failed to reconcile
- what resources belong to an application

This is valuable for production operations and incident response.

---

### 2. Continuous Reconciliation

Argo CD continuously compares the desired state in Git against the live Kubernetes state.

This supports:

- drift detection
- drift correction
- controlled synchronization
- safer production changes
- repeatable cluster recovery

---

### 3. App-of-Apps and ApplicationSet Support

The platform will grow across namespaces, clusters, and environments.

Argo CD supports scalable organization through:

- root applications
- app-of-apps
- ApplicationSets
- cluster generators
- Git directory generators
- environment-specific overlays

This fits the Infrastructure repository model well.

---

### 4. Git-Based Promotion

Promotion is performed through pull requests.

A production promotion should be visible as:

- a Git commit
- a pull request
- CI validation
- review approval
- Argo CD sync status
- Kubernetes rollout status

This creates a clean audit trail.

---

### 5. Progressive Delivery Alignment

Argo CD works well with **Argo Rollouts** for progressive delivery.

This supports future deployment strategies such as:

- canary deployments
- blue/green deployments
- traffic shifting
- metric-based promotion
- automated rollback

---

### 6. Disaster Recovery

If a cluster needs to be rebuilt, Argo CD can reconcile the cluster back toward the desired state from Git.

This improves recovery for:

- platform components
- namespaces
- applications
- policies
- observability resources
- ingress and certificate resources

---

## Consequences

### Positive

- Clear GitOps source of truth
- Reduced configuration drift
- Better operational visibility
- Stronger production audit trail
- Easier cluster rebuilds
- Cleaner promotion workflow
- Better support for multi-environment Kubernetes management
- Strong integration path with Argo Rollouts

---

### Costs / Risks

- Argo CD becomes a critical platform component
- broken manifests can block reconciliation
- incorrect sync policies can cause unwanted changes
- repository structure must remain disciplined
- cluster bootstrap process must be documented and repeatable
- secrets must be handled carefully and never committed to Git

---

### Mitigations

- enforce CI validation before merge
- protect staging and production branches
- require pull request approvals for production-impacting changes
- use clear Application boundaries
- use sync waves where ordering matters
- avoid enabling auto-prune globally without review
- document break-glass procedures
- use External Secrets or Vault-backed secret delivery
- alert on degraded or out-of-sync Argo CD applications

---

## Security and Compliance Implications

### Change Control

Argo CD supports Git-based change control.

Production changes should require:

- pull request review
- CI validation
- linked issue or operational reason where appropriate
- clear commit history
- visible Argo CD reconciliation status

---

### Least Privilege

Argo CD access should be scoped carefully.

Recommended controls:

- use SSO when available
- avoid broad admin access for normal users
- separate read-only and operator roles
- restrict production sync permissions
- use Kubernetes RBAC intentionally
- avoid storing long-lived credentials in plain Kubernetes secrets where possible

---

### Secret Handling

Secrets must not be stored directly in Git.

Approved patterns:

- External Secrets Operator with Vault
- Vault CSI where appropriate
- sealed/encrypted secret workflows only if explicitly approved by a separate ADR

Argo CD should reconcile references to secrets, not plaintext secret values.

---

### Evidence

The following can be used as evidence for audits or reviews:

- pull requests
- commit history
- CI logs
- Argo CD application history
- Argo CD sync status
- Kubernetes rollout status
- policy validation results

## Implementation Notes

### Bootstrap Model

Each cluster should be bootstrapped with a minimal Argo CD installation.

After Argo CD is available, it should take over reconciliation of platform resources.

Recommended bootstrap flow:

1. provision the cluster
2. install base prerequisites
3. install Argo CD
4. apply the root Argo CD application
5. allow Argo CD to reconcile platform and application resources
6. validate application health and sync status

---

### Recommended Repository Structure

High-level structure:

```text
Kubernetes/
  apps/
    prod/
    staging/
    dev/

  clusters/
    prod/
    staging/
    dev/

  platform/
    base/
    overlays/
````

Cluster bootstrap examples:

```text
Kubernetes/clusters/prod/
Kubernetes/clusters/staging/
Kubernetes/clusters/dev/
```

Application examples:

```text
Kubernetes/apps/prod/monitoring/
Kubernetes/apps/prod/security/
Kubernetes/apps/prod/sinless-games/
Kubernetes/apps/prod/istio-system/
```

---

### Argo CD Application Strategy

Use a root application per cluster.

Example pattern:

```text
root-app
  ├── platform
  ├── networking
  ├── security
  ├── monitoring
  ├── data
  └── applications
```

For larger groups of applications, use ApplicationSets where it reduces repetition.

---

### Sync Policy

Recommended default:

* manual sync for production-sensitive applications
* automated sync for low-risk platform support resources where appropriate
* prune enabled only where ownership is clear
* self-heal enabled where drift correction is desired
* sync waves for dependency ordering

Avoid enabling broad automatic pruning until the ownership boundaries are proven.

---

### Promotion Conventions

Promotion pull requests should:

* change only the intended version or configuration
* reference related issues, ADRs, or operational context
* include CI validation results
* avoid unrelated formatting changes
* include rollback notes when production-impacting

Promotion path:

```text
dev → staging → prod
```

---

### Drift and Break-Glass

Normal operations should not use manual `kubectl apply` for Git-managed resources.

Break-glass changes are allowed only when:

* the change is urgent
* the change is logged
* the change is time-limited
* the Git source of truth is updated afterward
* Argo CD reconciliation is restored afterward

Break-glass process:

1. identify the affected Argo CD application
2. pause or disable sync if needed
3. make the emergency change
4. document the reason
5. create a follow-up pull request
6. reconcile Git with the live state
7. re-enable normal Argo CD sync

---

## Validation

This ADR is successful when:

* a new cluster can be bootstrapped and reconciled from Git
* Argo CD reports application health and sync status
* environment overlays reconcile correctly
* drift is detected
* intentional drift is corrected or clearly reported
* dev → staging → prod promotion is pull-request based
* production changes are auditable
* critical platform components remain stable across reconciles

Suggested validation tests:

* change a namespace label manually and confirm Argo CD detects drift
* rebuild a dev cluster and confirm it converges from Git
* promote an application version from dev to staging
* validate sync waves for dependent platform components
* confirm Argo CD reports degraded applications correctly
* confirm alerts fire for degraded or out-of-sync production applications

---

## Rollback Plan

If Argo CD causes or worsens an incident:

1. identify the affected application
2. disable auto-sync for the affected application if enabled
3. revert the offending Git commit
4. sync the known-good version
5. verify workload health
6. document the incident and corrective action

If Argo CD itself is unhealthy:

1. inspect Argo CD controller, repo-server, Redis, and API server pods
2. verify Git repository access
3. verify Kubernetes API access
4. restore Argo CD from known-good manifests
5. manually apply only the minimum required recovery resources
6. return control to GitOps after recovery

A permanent switch away from Argo CD requires:

* a superseding ADR
* migration plan
* rollback strategy
* validation plan
* updated operational documentation
