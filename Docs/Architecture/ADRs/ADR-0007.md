## Context

The platform is designed to be operated via **Git pushes** with:

* reproducible cluster state
* minimal configuration drift
* auditable change control
* environment separation and promotion (Development → Staging → Production)

Kubernetes workloads and platform components (Istio, observability, Vault integration, cert-manager, etc.) must be deployed and managed declaratively.

The platform also requires:

* consistent structure for multi-cluster management
* progressive delivery workflows that can be driven by Git (not manual kubectl)
* a strong compliance story where:

  * every material change is reviewed
  * automation validates changes
  * evidence artifacts can be produced

A GitOps controller is necessary to continuously reconcile the desired state defined in Git with the running state of each cluster.

## Decision

Adopt **FluxCD** as the GitOps controller for all Kubernetes clusters.

FluxCD is responsible for:

* continuously reconciling each cluster’s desired state from the Infrastructure monorepo
* managing Kustomize overlays and Helm releases as code
* enabling promotion workflows by Git changes (PR-based)

Each cluster is bootstrapped from a dedicated path:

* `Kubernetes/Clusters/Development/`
* `Kubernetes/Clusters/Staging/`
* `Kubernetes/Clusters/Production/`

Platform components and applications are defined as:

* shared bases (common manifests/Helm)
* environment overlays (differences per cluster)

Promotion model:

* **Development → Staging → Production** is performed via PRs that promote versions and configuration through environment overlays.

## Scope

This ADR governs:

* selection of FluxCD as the GitOps controller
* repository-to-cluster reconciliation model
* high-level repo structure for cluster paths
* the principle that promotions happen through Git changes

This ADR does not:

* define the full repo layout (covered by ADR-0001 and implementation docs)
* choose specific progressive delivery mechanics (covered by ADR-0008)
* define CI pipelines (covered in implementation blueprint docs)

## Alternatives Considered

### A1) Argo CD

**Pros**

* strong UI and visualization
* large community

**Cons**

* different operational model than Flux
* progressive delivery integration choices differ
* preference is to align with Flux + Flagger ecosystem

### A2) CI-only deployments (kubectl apply/helm upgrade from GitHub Actions)

**Pros**

* simple initial setup

**Cons**

* not true continuous reconciliation
* drift can occur silently
* rollback and audit are less clean

### A3) “Manual ops” (kubectl by operators)

**Pros**

* immediate control

**Cons**

* high drift risk
* poor auditability
* not scalable

## Rationale

FluxCD is selected because it:

1. **Reconciles continuously**

   * Desired state is enforced, drift is corrected.

2. **Fits promotion workflows**

   * Git is the promotion mechanism; PRs become audit artifacts.

3. **Integrates with progressive delivery**

   * Flagger is Flux-native and aligns to canary requirements.

4. **Supports Git-centric compliance**

   * approvals, reviews, and CI artifacts map cleanly to change control expectations.

5. **Works well with Kustomize + Helm**

   * both can be used and standardized across environments.

## Consequences

### Positive

* Reduced drift and clearer operational truth
* Strong audit and change control story
* Repeatable cluster rebuild and disaster recovery

### Costs / Risks

* Requires disciplined repo hygiene (broken YAML breaks reconciliation)
* Git history becomes operationally significant (must protect branches)
* Flux controllers are critical cluster components

### Mitigations

* Enforce validation gates in CI:

  * YAML/schema validation
  * Kustomize build checks
  * Helm lint/template checks
  * policy-as-code checks

* Use protected branches and required approvals for staging/production

* Use per-environment reconciliation intervals and health checks

* Document break-glass procedure (rare, logged)

## Security & Compliance Implications

* **Least privilege:**

  * Flux uses scoped Kubernetes permissions.
  * Separate service accounts per namespace where feasible.

* **Change control:**

  * Staging/prod require PR approvals; commits are attributable.

* **Evidence:**

  * Promotion PRs + CI logs + Flux reconciliation status provide evidence.

* **Secret handling:**

  * Flux must not store secrets in Git.
  * Secrets are injected via Vault patterns (External Secrets Operator or CSI).

## Implementation Notes

### Bootstrap

* Each cluster is bootstrapped with FluxCD pointing at its cluster path.
* Bootstrap artifacts (deploy keys/tokens) are stored in Vault.

### Repo structure (high level)

Recommended:

* `Kubernetes/Base/` — shared components
* `Kubernetes/Apps/` — app definitions (HelmRelease/Kustomize)
* `Kubernetes/Clusters/<env>/` — per-cluster kustomizations and overlays

### Promotion conventions

* Prefer version pinning in one place:

  * Helm chart versions pinned in HelmRelease values
  * container versions pinned via image automation policies or explicit tags

* Promotion PRs should:

  * change only what’s being promoted
  * reference the related ADR(s)
  * include evidence from CI

### Drift and break-glass

* Normal operations forbid manual `kubectl apply` changes that diverge from Git.
* Break-glass changes are:

  * time-limited
  * documented
  * followed by a Git reconciliation fix

## Validation

This ADR is successful if:

* A cluster can be rebuilt from scratch and converges to desired state via Flux
* Drift is detected and corrected (or fails safe with visible alerts)
* Promotions dev → staging → prod are PR-based and reproducible
* Critical components (Istio, observability, cert-manager) remain stable across reconciles

Suggested test cases:

* intentionally drift a namespace label and confirm Flux reverts it
* rebuild dev cluster and validate it converges automatically
* promote an application version through overlays and validate behavior

## Rollback Plan

If Flux causes an incident:

* pause reconciliation for the affected kustomization/helmrelease
* revert the offending commit via PR
* resume reconciliation

A permanent switch away from Flux requires:

* a superseding ADR
* migration plan to an alternate GitOps controller
