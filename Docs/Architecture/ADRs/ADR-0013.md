## Context

The platform must support reliable recovery across multiple layers:

* Proxmox VMs and containers (CTs)
* Kubernetes cluster state and workloads
* persistent data used by platform services (GitLab, Vault, MinIO, Authentik, DNS)
* configuration and evidence artifacts stored in Git

Backups must satisfy operational and compliance requirements:

* predictable schedules
* encryption at rest and in transit
* retention policies
* restore testing (drills)
* audit evidence of backup success and recovery capability

You explicitly require:

* **Proxmox Backup Server (PBS)**
* **Velero** for Kubernetes backups
* **MinIO** as the local S3-compatible backend

The repo is the “source of truth” for desired state (ADR-0001). Backups must focus on:

* data and state that cannot be reconstructed from Git alone
* credentials and keys that must be preserved (stored in Vault)

## Decision

Adopt a two-layer backup and DR approach:

* **Proxmox Backup Server (PBS)** is the primary backup system for:

  * Proxmox VMs
  * Proxmox CTs
  * Proxmox host-level backup jobs and schedules

* **Velero** is the primary backup system for:

  * Kubernetes cluster resources (namespaces, CRDs, workloads)
  * Kubernetes persistent volume backups (as supported by CSI/snapshot capabilities)

* **MinIO** is the S3-compatible object storage backend for Velero backups.

Vault is the system of record for secrets and keys (ADR-0012). Backup credentials and encryption keys must be stored in Vault.

## Scope

This ADR governs:

* choice of PBS for VM/CT backups
* choice of Velero for Kubernetes backups
* use of MinIO as the Velero backend
* baseline requirements for retention, encryption, and restore testing

This ADR does not:

* define the final retention windows and schedules per environment (those are set in environment configuration)
* decide the final storage CSI/snapshot provider (that affects PV backups and is covered in separate ADRs)
* replace app-specific backup runbooks (GitLab, Vault, Authentik) beyond establishing the platform pattern

## Alternatives Considered

### A1) VM backups only (PBS only)

**Pros**

* simplest

**Cons**

* Kubernetes-native restoration is harder
* PV-level restores can be awkward
* cluster-level recovery is slower and less granular

### A2) Kubernetes backups only (Velero only)

**Pros**

* GitOps-aligned

**Cons**

* does not protect Proxmox host/VM state
* cannot easily restore VM-based systems outside K8s

### A3) Offsite cloud backup only

**Pros**

* strong protection against local disasters

**Cons**

* cost and bandwidth
* dependency on external provider
* requires careful multi-provider encryption and lifecycle control

### A4) Snapshot-only without backup retention

**Pros**

* fast and convenient

**Cons**

* snapshots are not backups
* retention and immutability are weak

## Rationale

PBS + Velero is selected because it:

1. **Separates concerns cleanly**

   * PBS covers hypervisor-level assets
   * Velero covers Kubernetes-native assets

2. **Improves recovery speed and granularity**

   * VM restores are fast and consistent
   * Kubernetes restores can be namespace-scoped and workload-scoped

3. **Aligns to GitOps**

   * Git reconstructs desired state; backups preserve data/state

4. **Supports compliance expectations**

   * defined schedules, retention, encryption, and drills

5. **Keeps storage local-first**

   * MinIO provides S3 compatibility without immediate cloud dependency

## Consequences

### Positive

* Strong layered recovery capability
* Clear operational ownership boundaries
* Better evidence for audits (job success, restore drills)

### Costs / Risks

* Backup sprawl if retention is not managed
* MinIO becomes a critical dependency for Velero
* PV backups depend on CSI snapshot support and correct configuration
* Application-consistent backups may require app-level hooks

### Mitigations

* Enforce retention policies per environment

* Treat MinIO as a Tier-1 service:

  * redundancy
  * backups of MinIO itself
  * monitoring

* Define app-level backup patterns:

  * GitLab exports/backups
  * Vault snapshots and unseal recovery
  * Authentik database backups

* Run restore drills on a schedule and store results in repo

## Security & Compliance Implications

* **Encryption**:

  * PBS backups must be encrypted (where supported) and transferred securely.
  * Velero backups should use encrypted S3 objects and secure credentials.

* **Least privilege**:

  * PBS backup users and API tokens scoped to required datastores.
  * MinIO access keys scoped to required buckets.

* **Audit evidence**:

  * Backup job logs and restore drill reports must be retained.

* **Immutability**:

  * Where feasible, enable immutable retention or WORM-like controls.

## Implementation Notes

### Proxmox Backup Server (PBS)

* PBS runs as a dedicated service (VM or hardware as available).

* Proxmox nodes are configured with PBS as a storage target.

* Backup jobs are defined as code:

  * which VMs/CTs
  * schedule
  * retention
  * encryption

* Recommended grouping:

  * separate backup jobs per environment or workload tier

### Velero

* Velero is deployed per Kubernetes cluster.

* Each cluster writes to a distinct MinIO bucket or prefix namespace:

  * `velero-development/`
  * `velero-staging/`
  * `velero-production/`

* Credentials for S3 access are stored in Vault and injected into the cluster.

### MinIO

* MinIO is deployed as a platform service (on pve-02/pve-03 intent).
* MinIO must have:

  * redundancy strategy (erasure coding or replication where feasible)
  * monitoring (availability, capacity, error rates)

### What to back up (baseline)

Hypervisor layer:

* all platform service VMs (Vault, Authentik, GitLab, MinIO, DNS)
* Kubernetes node VMs (optional; useful for fast restore but not required if cluster rebuild is easy)

Kubernetes layer:

* platform namespaces (observability, ingress/gateway, cert-manager, policy engines)
* application namespaces
* PVs where supported

### Restore drills

Minimum drill requirements:

* Quarterly:

  * restore a VM from PBS into an isolated network
  * validate boot + service availability

* Quarterly (staggered):

  * restore a Kubernetes namespace via Velero
  * validate application behavior

Drill outcomes must be documented and version-controlled.

## Validation

This ADR is successful if:

* PBS backups complete on schedule and retention is applied
* Velero backups complete and are restorable
* MinIO remains stable and monitored
* At least one full restore drill has been executed per environment tier

Suggested test cases:

* restore a GitLab VM and verify Git operations
* restore a test namespace and verify service functionality
* simulate MinIO bucket credential rotation and confirm Velero continues

## Rollback Plan

If Velero/MinIO is unstable initially:

* keep PBS as the authoritative backup for platform services
* reduce Velero scope to cluster resources only (no PV backups) until stable

If PBS is unstable:

* temporarily use local vzdump or Proxmox host-based backups and restore PBS

A permanent redesign requires a superseding ADR.

## Follow-ups

* Define final retention windows per environment and workload class
* Decide PV snapshot strategy (CSI provider support) and document
* Create app-specific backup runbooks (GitLab, Vault, Authentik, MinIO)
* Implement automated evidence generation for backup success and restore drills
