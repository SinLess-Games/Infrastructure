## Context

The platform must support reliable recovery across multiple infrastructure
layers.

The recovery scope includes:

- Proxmox virtual machines
- Proxmox containers
- Kubernetes cluster state
- Kubernetes workloads
- Kubernetes persistent volumes where supported
- platform service data
- backup metadata
- object storage data
- credentials and cryptographic material
- Git-based configuration and evidence artifacts

The infrastructure uses Git as the source of truth for desired state. Git can
reconstruct manifests, configuration, and platform intent. Backups must preserve
the data and runtime state that cannot be reconstructed from Git alone.

The platform requires:

- predictable backup schedules
- explicit retention policies
- encryption at rest where supported
- encryption in transit
- restore testing
- documented recovery procedures
- backup health monitoring
- audit evidence for successful backups and restore drills

The required backup systems are:

- **Proxmox Backup Server** for Proxmox VM and CT backups
- **Velero** for Kubernetes resource and namespace backups
- **Garage** as the self-hosted S3-compatible object storage backend for Velero

Garage replaces MinIO as the local S3-compatible object storage platform.

Vault remains the system of record for secrets, keys, and backup credentials.

---

## Decision

Adopt a layered backup and disaster recovery model.

The standard backup layers are:

| Layer | Tool | Purpose |
| --- | --- | --- |
| Hypervisor | Proxmox Backup Server | Back up Proxmox VMs and CTs |
| Kubernetes | Velero | Back up Kubernetes resources and supported persistent data |
| Object Storage | Garage | S3-compatible backup target for Velero and related backup workflows |
| Secrets | Vault | Store backup credentials, encryption keys, tokens, and recovery material |
| Source of Truth | Git | Store desired state, manifests, infrastructure code, and documentation |

---

## Primary Decision

Use **Proxmox Backup Server** as the authoritative backup system for Proxmox
assets.

Use **Velero** as the Kubernetes backup and restore system.

Use **Garage** as the S3-compatible object storage backend for Velero backups.

Use **Vault** for all backup-related credentials and sensitive recovery
material.

---

## Responsibilities

### Proxmox Backup Server

PBS is responsible for:

- Proxmox VM backups
- Proxmox CT backups
- host-level scheduled backup jobs
- VM and CT restore workflows
- backup retention at the hypervisor layer
- encrypted Proxmox backup repositories where configured

---

### Velero

Velero is responsible for:

- Kubernetes resource backups
- namespace-scoped restores
- cluster-scoped resource restores where appropriate
- CRD and workload metadata backups
- persistent volume backup workflows where supported
- restore drills for Kubernetes workloads

---

### Garage

Garage is responsible for:

- S3-compatible object storage for Velero backup objects
- durable local-first backup storage
- bucket-level separation between environments
- access key separation between clusters and workloads
- object storage monitoring and capacity planning

Garage is a backup target. It is not a replacement for Proxmox Backup Server,
Vault snapshots, database-native backups, or Git.

---

### Vault

Vault is responsible for:

- Garage access keys
- Velero S3 credentials
- PBS credentials where applicable
- encryption keys
- restore credentials
- break-glass recovery material
- backup automation tokens

Plaintext backup credentials must not be committed to Git.

---

## Scope

This ADR governs:

- use of PBS for VM and CT backups
- use of Velero for Kubernetes backups
- use of Garage as the Velero S3 backend
- high-level backup separation between Proxmox, Kubernetes, object storage, and secrets
- baseline restore testing expectations
- baseline retention and evidence requirements

This ADR does not:

- define the exact final retention windows for every workload
- define every Velero schedule
- define every PBS backup job
- define every app-specific backup runbook
- define the final CSI snapshot provider
- define the complete Garage deployment manifest
- define Longhorn backup implementation details
- replace database-native backup requirements

Application-specific backup and restore details must be documented separately.

---

## Non-Goals

This ADR does not claim that:

- snapshots alone are sufficient backups
- Git alone is sufficient disaster recovery
- Velero alone can restore every stateful workload
- PBS alone can provide Kubernetes-native recovery
- Garage should be the only copy of critical backups
- object storage availability is the same as tested recoverability

Backups are only trusted after successful restore testing.

## Alternatives Considered

### A1) Proxmox Backup Server Only

Use only PBS for all infrastructure recovery.

**Pros:**

- simple operational model
- strong VM and CT restore path
- good fit for Proxmox environments
- reliable for full-system rollback

**Cons:**

- Kubernetes-native restores are harder
- namespace-level restores are less convenient
- application migration between clusters is harder
- PV-level restore strategy is less flexible
- cluster rebuilds become more VM-image dependent

PBS-only recovery is rejected because Kubernetes workloads require
Kubernetes-native backup and restore workflows.

---

### A2) Velero Only

Use only Velero for backups.

**Pros:**

- Kubernetes-native
- aligns well with GitOps
- supports namespace-level restores
- useful for cluster migration

**Cons:**

- does not protect Proxmox hosts
- does not protect non-Kubernetes VMs
- does not back up VM-based platform services
- does not replace app-native database backups
- does not protect the object storage backend itself

Velero-only recovery is rejected because the platform includes critical
hypervisor-level and VM-level assets.

---

### A3) MinIO as the S3 Backend

Use MinIO as the self-hosted S3 backend for Velero.

**Pros:**

- common S3-compatible object storage option
- strong ecosystem awareness
- widely used with Kubernetes tooling

**Cons:**

- Garage has been selected for this platform
- MinIO is no longer the desired local object storage standard
- using both Garage and MinIO would increase operational sprawl
- bucket, credential, monitoring, and backup standards should converge on one local object store

MinIO is rejected for this platform in favor of Garage.

---

### A4) Cloud-Only Object Storage

Use only a cloud object store for Velero and other backup targets.

Examples:

- Cloudflare R2
- AWS S3
- Backblaze B2
- Wasabi

**Pros:**

- strong offsite disaster protection
- less local hardware dependency
- provider-managed durability
- useful for ransomware and site-loss scenarios

**Cons:**

- recurring cost
- bandwidth dependency
- restore speed depends on WAN
- credentials must be managed carefully
- provider outages or account issues become part of the recovery risk

Cloud-only object storage is rejected as the primary local backup target.

Cloud object storage may still be used as an offsite replication tier.

---

### A5) Snapshot-Only Strategy

Use only storage-level snapshots.

**Pros:**

- fast
- convenient
- useful for short-term rollback
- low operational friction

**Cons:**

- snapshots are not complete backups
- snapshots often share failure domains with primary data
- retention may be weak
- immutability may be weak
- accidental deletion or corruption can propagate
- restore evidence is often poor

Snapshot-only recovery is rejected.

Snapshots may be used as part of the backup strategy, but they do not replace
PBS, Velero, app-native backups, or offsite copies.

---

## Rationale

### 1. Layered Recovery

Different platform layers need different recovery tools.

PBS is strong for Proxmox assets.

Velero is strong for Kubernetes resources.

Garage provides the S3-compatible object storage layer needed by Velero and
other backup workflows.

Vault protects the sensitive credentials and keys required to restore systems.

This separation keeps recovery paths clear.

---

### 2. GitOps-Friendly Kubernetes Recovery

Git stores desired state, but Git does not store runtime data.

Velero complements GitOps by backing up cluster objects and restore metadata
that are useful during recovery.

A recovered cluster should be able to use both:

- Git to restore intended desired state
- Velero to restore selected Kubernetes runtime state and persistent data

---

### 3. Garage as Local S3-Compatible Backup Storage

Garage is selected as the local object storage standard.

Using Garage for Velero gives the platform:

- self-hosted S3-compatible storage
- local backup target control
- bucket-level separation
- integration with S3-compatible tooling
- a consistent object storage platform for future internal services

Garage should be treated as critical infrastructure when used for backups.

---

### 4. Clear Recovery Boundaries

The backup model avoids overloading one tool with every recovery responsibility.

PBS should restore VM and CT infrastructure.

Velero should restore Kubernetes resources.

Database-native backup tools should restore databases.

Vault recovery procedures should restore secrets and cryptographic material.

Garage should store backup objects and must itself be protected.

---

### 5. Audit and Evidence

The platform requires evidence that backups work.

Evidence should include:

- successful PBS job logs
- successful Velero backup records
- Garage bucket health and capacity metrics
- restore drill records
- backup failure alerts
- documented recovery test results

A backup that has never been restored is not considered proven.

---

## Consequences

### Positive

- strong VM and CT restore path through PBS
- Kubernetes-native restore path through Velero
- local S3-compatible backup backend through Garage
- clear separation of responsibilities
- better disaster recovery posture
- better audit evidence
- simpler long-term object storage standardization
- reduced dependency on MinIO

---

### Costs and Risks

- Garage becomes a critical backup dependency
- Garage itself must be backed up and monitored
- Velero restores depend on Garage availability
- backup storage can grow quickly
- retention misconfiguration can consume disk capacity
- PV backups depend on CSI and storage backend support
- application-consistent backups may require app-specific hooks
- restores can fail if CRDs, operators, or storage classes are missing
- local-only backup storage does not protect against full-site loss

---

### Mitigations

- treat Garage as Tier-1 infrastructure
- use at least a multi-node Garage deployment for important backup data
- monitor Garage availability, capacity, errors, and replication health
- protect Garage credentials in Vault
- separate buckets by environment and purpose
- use lifecycle and retention policies where supported by the workflow
- keep PBS backups for Garage nodes or Garage-hosting VMs
- add offsite replication for critical backup buckets
- test Velero restores on a schedule
- test PBS restores on a schedule
- document app-specific restore procedures
- alert on failed backup jobs
- alert on stale backups
- alert on low Garage capacity

---

## Security and Compliance Implications

### Encryption

Backup data must be protected in transit and at rest where supported.

Requirements:

- use TLS for Garage S3 endpoints exposed to clients
- use encrypted PBS repositories where supported
- protect Velero backup credentials in Vault
- protect Garage root/admin credentials in Vault
- rotate access keys on a defined schedule
- avoid plaintext credentials in manifests

---

### Least Privilege

Backup credentials must be scoped narrowly.

Garage access keys should be separated by:

- environment
- cluster
- workload class where appropriate
- backup purpose

Recommended bucket separation:

```text
velero-dev
velero-staging
velero-prod
longhorn-dev
longhorn-staging
longhorn-prod
app-backups-dev
app-backups-staging
app-backups-prod
```

Velero should only have access to the bucket or prefix it needs.

---

### Auditability

The following records should be retained:

- PBS backup job history
- Velero backup history
- Velero restore history
- Garage access key changes
- Garage bucket changes
- restore drill reports
- backup failure incidents
- backup-related pull requests

---

### Immutability

Where feasible, backups should be protected from accidental or malicious
deletion.

Possible controls:

- separate write credentials from administrative credentials
- restrict delete permissions where practical
- use offsite replicated copies for critical data
- keep PBS retention policies independent from Kubernetes backup retention
- avoid using the same credentials for backup writes and bucket administration

---

### Secret Handling

No backup credentials should be committed to Git.

Approved patterns:

- Vault to External Secrets Operator
- Vault CSI where appropriate
- sealed or encrypted secret workflows only if approved separately
- manual break-glass retrieval from Vault when required

---

## Implementation Notes

### Proxmox Backup Server

PBS runs as dedicated backup infrastructure.

PBS should back up:

- platform service VMs
- Kubernetes node VMs where useful
- Garage VMs or nodes if Garage is VM-based
- DNS infrastructure
- authentication infrastructure
- Git and automation infrastructure
- other critical Proxmox workloads

Recommended PBS job grouping:

- infrastructure-core
- kubernetes-control-plane
- kubernetes-workers
- platform-services
- storage-services
- security-services
- development-services

PBS backup jobs should define:

- included VMs and CTs
- schedule
- retention
- notification behavior
- encryption settings where supported
- restore priority

---

### Velero

Velero is deployed per Kubernetes cluster.

Each cluster writes to a dedicated Garage bucket.

Recommended buckets:

```text
velero-dev
velero-staging
velero-prod
```

Velero credentials are stored in Vault and injected into the cluster using the
approved secret delivery pattern.

Velero should back up:

- namespaces
- deployments
- services
- config maps
- CRDs where appropriate
- custom resources where appropriate
- ingress and gateway resources
- cert-manager resources
- External Secrets resources
- policy resources
- selected persistent volume data where supported

Velero backup schedules should be defined as code.

---

### Garage

Garage is the local S3-compatible backup target.

Garage should have:

- persistent metadata storage
- persistent data storage
- multi-node deployment for production backup data
- monitored capacity
- monitored availability
- monitored replication health
- documented recovery procedures
- documented key rotation procedures
- separate buckets per environment
- separate access keys per backup workflow

Garage must not be the only copy of its own recovery data.

When Garage stores critical backups, protect Garage with:

- PBS backups of Garage nodes or VMs
- documented Garage recovery steps
- offsite replication for critical backup buckets where feasible
- regular object listing and restore tests

---

### Garage Bucket Naming

Use simple, environment-scoped bucket names.

Recommended baseline:

```text
velero-dev
velero-staging
velero-prod
```

Optional future buckets:

```text
longhorn-dev
longhorn-staging
longhorn-prod
database-backups-dev
database-backups-staging
database-backups-prod
observability-archives-prod
```

Avoid sharing one bucket across all clusters.

---

### Garage Credentials

Use one access key per cluster and purpose.

Examples:

```text
velero-dev-writer
velero-staging-writer
velero-prod-writer
longhorn-prod-writer
```

Store the following in Vault:

- Garage endpoint
- Garage region
- bucket name
- access key ID
- secret access key
- TLS configuration where needed

Recommended Garage region value:

```text
garage
```

---

### Velero Garage Configuration

Velero should use Garage as an S3-compatible provider.

The exact flags depend on the deployed Velero version and plugin, but the
configuration should include:

```text
provider: aws
bucket: velero-prod
region: garage
s3Url: https://garage.example.internal
s3ForcePathStyle: true
```

Use path-style access unless the Garage endpoint and DNS are explicitly prepared
for virtual-hosted bucket access.

## Restore Strategy

### Proxmox Restore

Use PBS when restoring:

- entire VMs
- entire CTs
- platform service machines
- Garage nodes or VMs
- Kubernetes nodes when rebuilding from VM backup is faster than reprovisioning

---

### Kubernetes Restore

Use Velero when restoring:

- namespaces
- Kubernetes resources
- selected workloads
- selected persistent volumes
- environment-specific Kubernetes state

For GitOps-managed resources, restore carefully.

If Argo CD owns a resource, decide whether recovery should come from:

- Git reconciliation
- Velero restore
- both, in a controlled order

---

### Object Storage Restore

Garage recovery must be documented separately.

Garage restore procedures should include:

- restoring Garage node or VM from PBS
- validating Garage cluster health
- validating bucket availability
- validating access keys
- validating Velero can list backups
- validating Velero can restore a test namespace

---

## Restore Drill Requirements

Restore drills are mandatory.

Minimum schedule:

| Drill | Frequency | Purpose |
| --- | --- | --- |
| PBS VM restore | Quarterly | Prove VM/CT recovery works |
| Velero namespace restore | Quarterly | Prove Kubernetes restore works |
| Garage bucket access test | Monthly | Prove backup objects are reachable |
| Backup credential rotation test | Semi-annually | Prove key rotation does not break backups |
| Critical app restore | Semi-annually | Prove app-specific recovery works |

Drill results must be documented.

Recommended drill record fields:

- date
- operator
- system restored
- backup source
- restore target
- restore duration
- validation checks
- failures
- follow-up actions
- related pull request or issue

## Validation

This ADR is successful when:

- PBS backups complete on schedule
- PBS retention policies are applied
- Velero backups complete successfully
- Velero writes backups to Garage
- Garage remains healthy and monitored
- Velero can list backups from Garage
- a Velero namespace restore has been tested
- a PBS VM restore has been tested
- Garage credentials are stored in Vault
- backup failures alert operators
- restore drills are documented

Suggested validation tests:

- restore a VM from PBS into an isolated network
- restore a test namespace with Velero
- confirm Velero can read and write to the Garage bucket
- rotate a Garage access key and confirm Velero resumes successfully
- simulate a failed backup and confirm alerting
- confirm stale backup detection works
- confirm Garage capacity alerts work
- restore a Garage-hosting VM and validate bucket access
- confirm Argo CD does not overwrite restored state unexpectedly

## Rollback Plan

If Velero with Garage is unstable:

1. keep PBS as the authoritative backup layer for platform service VMs
2. reduce Velero scope to Kubernetes resources only
3. temporarily disable PV backups if they are failing
4. validate Garage credentials and endpoint configuration
5. test Velero backup and restore against a non-production namespace
6. re-enable broader Velero coverage after validation

If Garage is unstable:

1. pause new backup writes if needed
2. verify Garage cluster health
3. restore affected Garage nodes or VMs from PBS if required
4. validate bucket and object access
5. temporarily redirect new Velero backups to an alternate S3-compatible backend if available
6. restore Garage service before resuming normal backup schedules

If PBS is unstable:

1. investigate PBS datastore health
2. verify Proxmox connectivity to PBS
3. temporarily use Proxmox local backup jobs where necessary
4. restore PBS service from known-good backup if needed
5. validate at least one VM restore after recovery

A permanent redesign requires:

- a superseding ADR
- migration plan
- restore validation plan
- updated runbooks
- updated architecture documentation

---

## Operational Guardrails

Critical workloads must not be considered protected until they have:

- a defined backup source
- a defined backup schedule
- a defined retention policy
- a documented restore process
- a successful restore test
- alerting for failed backups
- an owner

Backups must not depend on undocumented manual knowledge.
