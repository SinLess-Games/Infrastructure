## Context

The platform requires shared, resilient storage to support:

* High-availability VMs and platform services
* Rapid VM provisioning from templates
* Consistent VM mobility across Proxmox nodes
* Clear performance and reliability guarantees per workload class
* A compliance-ready posture (encryption options, auditability, predictable retention)

The Proxmox cluster contains nodes with different capabilities:

* **10Gb-capable:** pve-01, pve-04, pve-05
* **Service-heavy nodes:** pve-02 and pve-03 host MinIO, Vault, GitLab and will also provide OSD capacity
* **OSD grouping requirement:** pve-02 and pve-03 have OSDs that must be organized into explicit storage groups

Ceph is a strong fit because it can provide:

* replication and self-healing
* pool-level policy controls
* per-workload performance segmentation (by pool and device class)
* integration with Proxmox’s storage model (VM disks, templates, migrations)

Ceph also adds operational responsibility:

* network design must avoid mixing replication traffic with management and DMZ
* health must be monitored and acted on quickly
* upgrades require planning and testing

## Decision

Use **Proxmox-integrated Ceph** as the primary shared storage backend.

Implement explicit **OSD groups** (device classes) and map them to **distinct pools** that represent storage “classes” for workloads.

### Storage classes (pool intent)

At minimum, implement two workload classes:

* **Fast / latency-sensitive** (example: `ceph-ssd-fast`)

  * intended for: control-plane VMs, databases, Vault storage, GitLab metadata, etc.
  * device type: SSD where available

* **Bulk / capacity-oriented** (example: `ceph-hdd-bulk`)

  * intended for: general VM disks, archives, backups staging, less latency-sensitive services
  * device type: HDD where available

The naming above is conceptual; the authoritative names are defined in code.

### Ceph network separation

Ceph traffic is separated per ADR-0003:

* **VLAN 40 (Storage):** Ceph replication / cluster network
* **Client/public access network:** VLAN 20 and/or VLAN 30 depending on consumer

### Service placement expectations

* Tier-0/Tier-1 services (MinIO, Vault, GitLab) may be backed by Ceph pools selected for their class.
* Node-local storage remains acceptable for specific cases:

  * scratch space
  * dedicated high-performance ephemeral disks
  * workloads that must remain isolated to a specific node

## Scope

This ADR governs:

* Selection of Proxmox-integrated Ceph as the default shared storage backend
* The requirement to group OSDs into explicit storage classes and map to pools
* The requirement to isolate Ceph replication traffic on VLAN 40
* High-level pool intent and workload mapping

This ADR does not:

* Define final CRUSH rules and exact pool parameters (replication size, min_size, PG counts)
* Decide whether Kubernetes consumes Ceph directly via CSI (that should be its own ADR)
* Replace platform-specific runbooks (Ceph recovery procedures belong under `Docs/Operations/`)

## Alternatives Considered

### A1) ZFS local storage only + replication

**Pros**

* Great local performance
* Familiar admin model

**Cons**

* VM mobility is harder and less seamless
* Replication management becomes operationally heavy
* Storage HA is not as automatic as Ceph

### A2) External NAS/SAN appliance

**Pros**

* Centralized storage
* Often simpler day-to-day interface

**Cons**

* Potential single performance choke point
* Vendor constraints and feature gaps
* More difficult to keep “as code” within desired repo-driven automation

### A3) Rook-Ceph inside Kubernetes as primary storage

**Pros**

* Kubernetes-native provisioning and lifecycle

**Cons**

* Running storage inside k8s increases complexity
* Requires careful separation of “platform depends on storage” cycles
* Would likely result in “two Ceph clusters” (Proxmox + Rook), which is heavy

### A4) Mix of local disks + selective NFS/SMB

**Pros**

* Simplest for non-critical workloads

**Cons**

* Fragmented storage story
* Harder to guarantee performance and HA
* Compliance and audit become more difficult

## Rationale

Proxmox-integrated Ceph is selected because it:

1. **Enables VM mobility and HA operations**

   * Shared storage supports migrations and clustered operations.

2. **Provides policy controls at the pool level**

   * Replication, failure domain, and class mapping can be expressed per pool.

3. **Supports performance segmentation**

   * OSD groups and pools allow “fast” and “bulk” classes.

4. **Aligns with automation goals**

   * Proxmox-integrated Ceph can be provisioned and managed consistently via automation.

## Consequences

### Positive

* Predictable shared storage for critical workloads
* Better resilience and self-healing characteristics
* Clear path to storage class separation and capacity planning

### Costs / Risks

* Ceph is operationally complex compared to single-node storage.
* Misconfiguration can cause performance issues.
* Hardware variance (10Gb vs 1Gb) can create hotspots.

### Mitigations

* Enforce VLAN 40 isolation and prefer 10Gb nodes for heavy replication

* Standardize pool definitions in code (version-controlled)

* Maintain runbooks for:

  * degraded PG remediation
  * OSD replacement
  * node outage recovery
  * upgrade procedure

* Monitor aggressively (see ADR-0014)

## Security & Compliance Implications

* **Segmentation** of storage traffic reduces exposure and lateral movement.

* **Encryption at rest** options:

  * Ceph can be combined with disk-level encryption (LUKS) on OSD devices where feasible.
  * Pool-level encryption patterns depend on device and operational constraints.

* **Access control**:

  * Ceph credentials and client keys must be treated as secrets and stored in Vault.
  * Least privilege client keys (per consumer) should be preferred.

* **Evidence**:

  * pool definitions, replication settings, and changes are in Git
  * health dashboards and incident logs support audit requirements

## Implementation Notes

### OSD grouping

The intent is to explicitly label/group OSD devices so that:

* SSD-backed OSDs feed “fast” pools
* HDD-backed OSDs feed “bulk” pools

On **pve-02** and **pve-03**, OSDs must be assigned to their intended class. The exact approach depends on device composition and is implemented via automation.

### Pools and placement

Recommended patterns:

* Use distinct pools per storage class
* Prefer host-level failure domains (replication across nodes)
* Avoid mixing “fast” and “bulk” workloads in the same pool

### Proxmox storage definitions

Proxmox should define Ceph storage entries per pool, allowing operators and automation to select:

* `ceph-ssd-fast` storage for latency-sensitive VM disks
* `ceph-hdd-bulk` storage for general capacity VM disks

### Integration with backups

* PBS backs up VMs regardless of Ceph pool placement.
* For critical services, document whether:

  * VM-level backup is sufficient, or
  * application-level backups are required (GitLab, Vault, MinIO)

### Observability hooks

At minimum:

* Ceph health dashboards (overall health, PG state, OSD status)
* Capacity dashboards per pool
* Latency and throughput dashboards (per pool, per host)
* Alerts for:

  * degraded/peering PGs
  * OSD down/out
  * nearfull/full
  * slow ops

## Validation

This ADR is successful if:

* Ceph remains healthy under normal operations and maintenance
* VM migration between Proxmox nodes works for Ceph-backed disks
* Pool-level separation is observable (fast pool behaves predictably vs bulk)
* OSD failures can be handled with documented runbooks
* Backups complete successfully and restore drills are performed

Suggested test cases:

* Create VMs on each pool and verify performance separation
* Simulate an OSD failure and validate recovery procedure
* Reboot a Ceph node and confirm cluster health returns to OK

## Rollback Plan

If Ceph introduces unacceptable instability:

* Move critical VMs to node-local storage temporarily
* Reduce Ceph usage to non-critical workloads while stabilizing

A permanent change away from Ceph requires:

* a superseding ADR
* a migration plan for VM disks and service storage

## Follow-ups

* Define exact pool parameters and CRUSH rule approach in code
* Create Ceph maintenance and incident runbooks in `Docs/Operations/`
* Decide whether Kubernetes will consume Ceph directly (CSI) and document in a dedicated ADR
* Define encryption-at-rest standard (LUKS vs pool-level patterns) per hardware feasibility
