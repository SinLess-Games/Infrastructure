## Context

The platform targets a high-assurance posture aligned with:

* HIPAA
* SOC 2
* EU privacy regulations
* defense-grade expectations (least privilege, segmentation, auditability)

To meet these requirements consistently, controls must be enforced as code across:

* Kubernetes deployments
* Infrastructure as Code (Terraform)
* Host configuration automation (Ansible)
* CI pipelines and promotion workflows

Failure modes without policy-as-code:

* insecure workloads deployed accidentally (privileged pods, hostPath, no probes)
* missing encryption settings in Terraform resources
* excessive privileges and broad network access
* drift in security posture over time
* weak audit evidence (“we intended to be secure”) versus actual enforced controls

Requirements:

* Policies must be version-controlled and reviewed.

* Enforcement must be layered:

  * pre-merge CI checks
  * admission control in-cluster
  * periodic audits and drift detection

* Different environments require different strictness:

  * Dev allows faster iteration with guardrails
  * Staging mirrors production
  * Production is strict and blocks unsafe changes

## Decision (Proposed)

Adopt a layered policy-as-code enforcement approach composed of:

1. **Kubernetes admission control** (cluster enforcement)
2. **IaC scanning and guardrails** (Terraform validation)
3. **CI gates and evidence generation** (GitHub Actions)

### Kubernetes admission

**Primary tool (proposed):** Kyverno

* Reasoning:

  * Kubernetes-native policies
  * good ergonomics for mutation + validation
  * strong fit for GitOps workflows

**Secondary/optional:** OPA Gatekeeper

* Used where Rego expressiveness is required.

Policy model:

* Baseline policies apply to all clusters.
* Production policies are strict; staging closely mirrors production.
* Exceptions are explicit, time-bounded, and require approvals.

### IaC scanning and Terraform guardrails

Use CI-based scanning plus optional runtime guardrails:

* **TFLint** (lint and best practices)
* **tfsec** and/or **Checkov** (security posture scanning)
* **Trivy** (IaC + container config scanning where relevant)

Optionally:

* **Open Policy Agent** evaluation in CI for Terraform plan JSON to enforce organizational rules.

### CI gates

CI is the control-plane for change control:

* Policies must run on every PR.
* Production merges require:

  * passing policy checks
  * required reviewers
  * stored evidence artifacts (plans, reports)

## Scope

This ADR governs:

* the platform-wide policy-as-code strategy
* tool choices (Kyverno-first, Gatekeeper optional)
* CI enforcement structure

This ADR does not:

* enumerate every required policy rule (those are maintained as policy catalogs)
* define incident response procedures (separate ops docs)

## Alternatives Considered

### A1) OPA Gatekeeper only

**Pros**

* very expressive policy language
* common in enterprises

**Cons**

* steeper learning curve
* more friction for mutation policies

### A2) Kyverno only

**Pros**

* Kubernetes-native
* good DX

**Cons**

* some complex constraints may be easier in Rego

### A3) CI-only scanning (no admission control)

**Pros**

* simpler cluster footprint

**Cons**

* drift and manual changes can bypass controls
* runtime enforcement is weaker

### A4) Manual reviews only

**Pros**

* low tool overhead

**Cons**

* not scalable
* inconsistent enforcement

## Rationale

The layered approach is selected because it:

1. **Blocks unsafe changes before merge**

   * CI fails fast.

2. **Prevents unsafe workloads at runtime**

   * Admission control rejects invalid manifests.

3. **Produces audit evidence automatically**

   * Reports and plans become artifacts.

4. **Supports different strictness per environment**

   * Dev can iterate without weakening production.

5. **Fits GitOps**

   * Policies are declared and reconciled like any other resource.

## Consequences

### Positive

* Strong baseline security posture enforced consistently
* Reduced accidental insecure deployments
* Better compliance evidence and audit readiness

### Costs / Risks

* Policy complexity can slow teams if not managed carefully
* False positives can create frustration
* Exceptions can become “policy debt” if not controlled

### Mitigations

* Start with a well-defined baseline catalog.

* Enforce exceptions as explicit resources with:

  * expiry
  * justification
  * owner

* Use phased adoption:

  * warn mode in dev
  * enforce in staging
  * strict enforce in production

* Maintain a policy review cadence.

## Security & Compliance Implications

Policy-as-code is a primary mechanism for satisfying:

* least privilege
* encryption requirements
* segmentation and network controls
* change control and audit evidence
* vulnerability and configuration management

Evidence artifacts:

* CI reports for each PR
* signed/attested build metadata (where applicable)
* cluster audit events when policies block deployments

    * reusable GitHub Actions workflows and rules

### Kubernetes baseline policy catalog (intent)

Baseline policies typically include:

* require namespaces and standard labels

* disallow privileged containers

* disallow hostPID/hostNetwork unless explicitly allowed

* restrict hostPath mounts

* require:

  * readiness/liveness probes
  * resource requests/limits
  * non-root user

* enforce image provenance:

  * disallow `:latest`
  * require signed images (future phase)

* enforce network policy presence for sensitive namespaces

### Terraform baseline catalog (intent)

* enforce encryption at rest where supported
* require tagging/labels for ownership and environment
* disallow overly-open security groups
* require least privilege IAM policies

### Enforcement modes

* Development:

  * policy warnings where feasible

* Staging:

  * enforcement mirroring production as closely as possible

* Production:

  * strict enforcement
  * exceptions require explicit approval and expiry

### Operational guardrails

* Define a break-glass process:

  * used only for incidents
  * audited
  * reconciled back into policy

## Validation

This ADR is successful if:

* Unsafe Kubernetes manifests are blocked at admission.
* IaC changes that violate policy fail CI.
* Production policies are stricter than dev.
* Exceptions are rare, reviewed, and expire.

Suggested test cases:

* attempt to deploy a privileged pod and confirm rejection
* open a wide security group in Terraform and confirm CI failure
* deploy an app without resource limits and confirm policy action

## Rollback Plan

If enforcement causes unacceptable friction:

* switch dev to warn mode and refine rules
* keep staging/prod enforcement strict

If a policy causes an outage:

* apply a time-bounded exception
* open a follow-up task to remove the exception

A full removal of policy-as-code requires a superseding ADR.

## Follow-ups

* Confirm primary admission controller selection (Kyverno-first)
* Define the initial baseline policy catalog and implement it as code
* Add CI evidence artifact publishing and retention
* Define exception workflows and templates
