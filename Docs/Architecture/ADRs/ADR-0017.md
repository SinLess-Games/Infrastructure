## Context

The platform requires a self-hosted source control and CI system with:

* strong auditability and traceability
* integration into GitOps workflows
* support for multiple domains
* alignment with Zero Trust posture (SSO, MFA, no public management ports)
* reliable backup and restore capability

GitHub remains the authoritative remote for the Infrastructure monorepo, but GitLab is required internally to support:

* self-hosted CI execution
* internal repos/projects
* runner fleets for build pipelines
* potential artifact registries and internal automation

You also specified intended placement constraints:

* **pve-02** and **pve-03** will be used for **MinIO, Vault, GitLab** (service-heavy nodes)

GitLab is a Tier-1 service (high impact if down). It must be deployed with careful consideration of:

* data storage (Postgres, Gitaly, artifacts)
* performance (IOPS, latency)
* backups and DR
* secure access and auditing

## Decision

Deploy **GitLab self-hosted** as a platform service with the following operating model:

1. **Primary placement**

   * GitLab runs on infrastructure hosted primarily on **pve-02** and **pve-03**.
   * GitLab is deployed as:

     * a dedicated VM (recommended for early stability), with a path to Kubernetes deployment once the platform is mature, OR
     * a Kubernetes deployment from day one if operational confidence exists.

   The authoritative initial mode is selected by implementation, but the placement intent is fixed: pve-02/pve-03 are the compute/storage anchors.

2. **Access model**

   * GitLab is exposed externally only through **Cloudflare Tunnel + Access** when external access is required.
   * Internal access is available over the Infra/Services plane, with management access restricted.

3. **Identity**

   * GitLab uses **Authentik OIDC** for SSO.
   * Local admin accounts exist only for break-glass and are stored/controlled per Vault policy.

4. **Storage**

   * GitLab data is stored on a defined storage class per ADR-0005:

     * prefer “fast” Ceph pool for Postgres/Gitaly where feasible
     * artifacts can use capacity pools or object storage

   * GitLab object storage (artifacts, uploads, packages) uses **MinIO** S3 where feasible.

5. **Backup & DR**

   * GitLab is protected by:

     * VM-level backup to **Proxmox Backup Server (PBS)**, and
     * application-aware GitLab backups (exports + database backups)

   * Backup credentials and encryption keys are stored in Vault.

6. **Observability**

   * GitLab metrics/logs are integrated into the Grafana observability stack.

7. **Multi-domain support**

   * GitLab supports multiple domains by:

     * explicit hostname configuration
     * cert-manager issued certificates (or Cloudflare edge certificates) per ADR-0010/ADR-0011

## Scope

This ADR governs:

* where GitLab is placed (pve-02/pve-03 intent)
* the security and access operating model (SSO + tunnel-first)
* storage and backup expectations
* integration with MinIO, Vault, and observability

This ADR does not:

* decide GitLab edition/licensing or feature scope
* define exact runner topology (separate ADR recommended)
* define registry strategy (GitLab registry vs external registry)

## Alternatives Considered

### A1) Use GitHub only (no GitLab)

**Pros**

* simplest
* fewer services to operate

**Cons**

* less internal autonomy
* runner fleet control and internal CI requirements may be constrained

### A2) Run GitLab in Kubernetes only

**Pros**

* GitOps-native
* easier horizontal scaling in theory

**Cons**

* complex stateful components
* higher initial operational risk

### A3) Run GitLab as a dedicated VM only

**Pros**

* simplest stable self-host pattern
* clear backup model

**Cons**

* scaling and upgrades are more manual
* runners and integrations may be less elegant

### A4) Use a lighter forge (Gitea/Forgejo)

**Pros**

* lower resource footprint
* simpler ops

**Cons**

* missing GitLab’s CI depth and enterprise features

## Rationale

GitLab is selected and placed as described because it:

1. **Provides robust self-hosted CI**

   * supports internal pipelines and runner fleets.

2. **Aligns with Zero Trust and compliance**

   * OIDC SSO, MFA, audit logs, controlled access.

3. **Fits the platform service placement model**

   * pve-02/pve-03 are designated for core services.

4. **Supports a phased maturity approach**

   * VM-first stability with a path to Kubernetes-native operation.

5. **Integrates cleanly with MinIO and Vault**

   * object storage for artifacts, Vault for secrets.

## Consequences

### Positive

* Internal autonomy for CI and development workflows
* Strong auditability and workflow traceability
* Reduced reliance on external services for build execution

### Costs / Risks

* GitLab is operationally heavy (CPU/RAM/IO)
* Stateful complexity (Postgres, Redis, Gitaly)
* Requires disciplined backups and upgrades

### Mitigations

* Start with VM deployment for stability unless there is strong confidence in K8s stateful ops.
* Use MinIO object storage to reduce local disk pressure.
* Ensure “fast” storage class for latency-sensitive data.
* Implement strict monitoring and capacity planning.

## Security & Compliance Implications

* **Identity**:

  * OIDC SSO via Authentik with MFA.

* **Access**:

  * tunnel-first external access; no direct inbound exposure.

* **Audit**:

  * GitLab audit events and logs are retained.

* **Secrets**:

  * runner tokens, integration secrets, and encryption keys stored in Vault.

* **Data protection**:

  * backups to PBS and app-level backups provide layered recovery.

## Implementation Notes

### Deployment mode selection

Recommended maturity path:

1. VM-based GitLab (Omnibus or similar) for initial stability
2. Evaluate Kubernetes deployment once:

   * storage and backup patterns are proven
   * cluster reliability is high
   * observability and incident runbooks exist

### Storage layout intent

* Postgres:

  * fast pool / low latency

* Gitaly repositories:

  * fast pool

* Artifacts/uploads/packages:

  * MinIO S3 where feasible

### Backup model

* VM snapshot/backups via PBS

* GitLab native backups:

  * scheduled exports
  * database dumps

* Store backup schedules and retention as code.

### Observability

Collect:

* GitLab service metrics
* Postgres metrics
* Redis metrics
* logs

Alert on:

* high queue backlog
* database latency
* disk usage
* error rates

### Multi-domain

* Hostname(s) are explicitly defined per environment.
* Certificates are issued and rotated via cert-manager/Cloudflare.

## Validation

This ADR is successful if:

* GitLab is reachable internally, and externally only via Cloudflare Access when enabled.
* OIDC SSO works and roles map correctly.
* Artifacts can be stored and retrieved from MinIO.
* Backups succeed and restores are tested.

Suggested test cases:

* create a test project and run a pipeline with internal runners
* restore GitLab from PBS backup into isolated network
* restore from GitLab application backup and validate repositories

## Rollback Plan

If GitLab becomes unstable:

* reduce scope to critical repos only
* restore to previous known-good snapshot
* temporarily route CI to GitHub Actions if required

A long-term platform shift requires a superseding ADR.

## Follow-ups

* Create an ADR for runner fleet placement and security model
* Decide registry strategy (GitLab registry vs external) and document
* Define capacity planning and scaling thresholds for GitLab
* Add runbooks for upgrades and incident response
