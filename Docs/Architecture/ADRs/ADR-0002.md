## Context

The infrastructure platform uses **Proxmox VE** as the hypervisor layer and must support:

* Highly-available virtualization operations (maintenance, failures, migrations)
* Shared storage for VM mobility and critical services
* A predictable platform for provisioning Kubernetes clusters (VM-first)
* Segmented networks aligned to Zero Trust (management, storage, DMZ)
* A service substrate for Tier-0/Tier-1 components (Vault, Authentik, MinIO, GitLab, DNS)

The environment already includes the following nodes and operational intent:

* **pve-01, pve-02, pve-03, pve-04, pve-05** exist and will be cluster members.
* **10Gb-capable nodes:** pve-01, pve-04, pve-05
* **Service-focused nodes:** pve-02 and pve-03 will host key platform services:

  * MinIO (local S3)
  * Vault
  * GitLab + Runner
  * Technitium DNS
  * Proxmox Backup Server (PBS)
  * Unifi Controller (network mgmt)

Storage requirements:

* **Ceph (Proxmox-integrated)** is the default shared storage model.
* **pve-02 and pve-03** contain OSDs that must be organized into explicit storage groups (pool mapping).

Operational requirements:

* Proxmox UI must remain reachable during node maintenance and failures.
* Proxmox and storage health must be observable and auditable.
* Backups must be enforceable and testable.

## Decision

Create and maintain a single Proxmox VE cluster:

* **Cluster Name:** SinLess-Games
* **Nodes:** pve-01, pve-02, pve-03, pve-04, pve-05

Assign roles by intent (without hard-coding exclusivity):

* **Ceph / 10Gb-heavy nodes:** pve-01, pve-04, pve-05

  * Prefer Ceph MON/MGR placement and high-bandwidth storage traffic here
  * Prefer storage replication and heavy VM disk IO here

* **Platform services nodes:** pve-02, pve-03

  * Host Tier-0/Tier-1 services (MinIO, Vault, GitLab, Technitium DNS, etc.)
  * Also host Ceph OSDs grouped into defined pools (see [ADR-0005](./ADR-0005.md))

Provide an HA access entrypoint for the Proxmox UI:

* Keepalived VIP + HAProxy with round robin (see [ADR-0004](./ADR-0004.md))

## Scope

This ADR governs:

* Proxmox cluster membership and naming conventions
* High-level node role intent (10Gb/Ceph emphasis vs services emphasis)
* The baseline expectation that Proxmox is the virtualization substrate for Kubernetes and platform services

This ADR does not:

* Specify exact VM placement rules (that is environment and workload specific)
* Define the full Ceph pool layout (see ADR-0005)
* Define networking bridge/bond details (see ADR-0003)

## Alternatives Considered

### A1) 3-node Proxmox cluster only (minimal cluster)

**Pros**

* Simpler quorum management
* Smaller blast radius for cluster-wide configuration errors
* Lower complexity for initial Ceph deployment

**Cons**

* Underutilizes available hardware
* Less flexibility for service isolation and capacity planning
* Reduced headroom for HA + maintenance windows

### A2) Two Proxmox clusters (prod vs non-prod split)

**Pros**

* Stronger environment isolation
* Separate failure domains (non-prod can’t impact prod)

**Cons**

* Doubles operational overhead (two clusters to patch and manage)
* Harder to share VM templates and base images cleanly
* More complicated storage strategy and resource distribution

### A3) Proxmox-only for virtualization; run core services on separate bare-metal

**Pros**

* Fewer “eggs in one basket”
* Potentially simpler failure-domain story

**Cons**

* Requires additional physical hosts or operational complexity
* Breaks “single substrate” automation story
* Increases wiring, lifecycle, and drift risk

## Rationale

A single 5-node cluster provides the best balance:

1. **Capacity and flexibility**

   * Enough nodes to host both workloads and core platform services with isolation.

2. **Operational simplicity**

   * One cluster to manage, patch, template, and automate.

3. **Resilience**

   * HA patterns (UI VIP, Ceph, service placement) can be layered without managing multiple clusters.

4. **Clear physical intent**

   * 10Gb nodes focus on high-throughput storage and IO.
   * pve-02/pve-03 provide a stable home for Tier-0/Tier-1 services.

## Consequences

### Positive

* Simplified management and consistent cluster-wide configuration
* Improved VM mobility and maintenance workflows
* Clear role intent makes troubleshooting and capacity planning easier

### Costs / Risks

* A misconfiguration at the cluster layer can impact all environments.
* Service concentration on pve-02/pve-03 must be mitigated with:

  * HA placement (where feasible)
  * backups
  * restore drills

### Mitigations

* Enforce strict VLAN plane separation and default-deny firewalling (ADR-0003)
* Use HA UI entrypoint (ADR-0004)
* Enforce backups and restore drills (ADR-0013)
* Implement strong RBAC via OIDC + least privilege (ADR-0009)

## Security & Compliance Implications

* **Segmentation:**

  * Management access must be constrained to WireGuard overlay and/or Cloudflare Access.

* **Auditability:**

  * Cluster membership and configuration changes are made via code and tracked via PRs.

* **Availability evidence:**

  * Regular failover and restore drills generate audit evidence.

* **Least privilege:**

  * Proxmox RBAC must map to Authentik groups and roles.

## Implementation Notes

### Naming and identity

* Node names are stable: `pve-01` … `pve-05`
* Cluster name is stable: `SinLess-Games`

### Baseline configuration expectations

* NTP/time sync configured across all nodes
* Cluster firewall baseline applied (default deny across planes; explicit allow lists)
* SSH hardening and restricted administrative access

### Storage intent

* Ceph is integrated with Proxmox
* pve-02/pve-03 OSDs map into explicit pools/storage groups

### Service placement intent

* Vault, MinIO, GitLab, DNS are primarily placed on pve-02 and pve-03
* If service HA requires a third instance, preferentially use a 10Gb node VM as the tie-breaker/third placement

### Backups intent

* Proxmox Backup Server protects VMs/CTs
* Restore drills must include:

  * a VM restore from PBS
  * a service restore for at least one Tier-1 app

## Validation

This ADR is successful if:

* All five nodes join and remain healthy members of the `SinLess-Games` cluster

* A single node failure does not remove administrative access (VIP failover works)

* VM migrations between nodes succeed (subject to storage availability)

* Observability dashboards show:

  * node resource health
  * Ceph cluster health
  * VM inventory and status

* Backup policies are operational and at least one restore drill has been executed and documented

## Rollback Plan

If the 5-node cluster introduces unacceptable risk during early stabilization:

* Temporarily operate as a 3-node functional subset (e.g., pve-01/pve-02/pve-03) while:

  * preserving the naming scheme
  * keeping automation consistent

* Re-add pve-04/pve-05 after root cause is resolved.

If the single-cluster approach becomes a limiting factor long-term:

* Create a new ADR to **supersede** this one and define a split-cluster plan (prod vs non-prod) with a migration runbook.

## Follow-ups

* Define `resource pools` and placement guidelines per environment
* Finalize Ceph pool and CRUSH rule strategy (ADR-0005)
* Define a documented maintenance workflow (patching, reboot order, failover checks)
