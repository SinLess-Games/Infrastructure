## Context

The platform requires a Zero Trust access model that explicitly avoids:

* exposing management ports to the public Internet
* relying on fragile perimeter-only trust assumptions

External access is still required for selected services (e.g., portals, dashboards, GitLab, and possibly limited admin surfaces). The platform must support:

* multiple domain names
* consistent access policy enforcement (MFA, identity, device posture where available)
* auditability of access decisions
* operations even when networks change (IP changes, ISP changes)

Additionally:

* Cert issuance must work without opening inbound ACME endpoints (ADR-0010).

Cloudflare Tunnel and Access provide a model where services can be published without opening inbound firewall ports and can enforce identity-driven access policies.

## Decision

Adopt **Cloudflare Tunnel** as the primary mechanism for publishing external-facing service endpoints.

Adopt **Cloudflare Access** as the policy enforcement layer for externally reachable services.

Key rules:

* **No public inbound ports** are opened to management plane services.

* External access is allowed only through:

  * Cloudflare Tunnel
  * Cloudflare Access policies

* Cloudflare Access integrates with the identity plane (Authentik OIDC) for authentication.

## Scope

This ADR governs:

* the external access pattern for services
* the decision to use Cloudflare Tunnel and Access
* the rule that inbound ports are not opened for management services

This ADR does not:

* decide which services are permitted external exposure (that is controlled by policy and environment overlays)
* define internal-only access paths (WireGuard covers internal management access)

## Alternatives Considered

### A1) Traditional public ingress (open 80/443 to a reverse proxy)

**Pros**

* common and well-understood

**Cons**

* increases attack surface
* requires perimeter hardening and constant patch vigilance
* conflicts with “no inbound management ports” posture

### A2) Public VPN gateway only

**Pros**

* strong control if managed well

**Cons**

* VPN scaling and UX can be worse for broad access
* still exposes a public endpoint (VPN)

### A3) Self-hosted zero trust gateway (e.g., reverse proxy + OIDC + WAF)

**Pros**

* full control

**Cons**

* more operational overhead
* harder to match Cloudflare’s edge features and reliability

### A4) Tailscale-based access model

**Pros**

* very convenient mesh access

**Cons**

* introduces another identity and policy domain
* may not align with “Cloudflare-first” DNS/cert posture

## Rationale

Cloudflare Tunnel + Access is selected because it:

1. **Eliminates inbound ports**

   * services initiate outbound tunnels to Cloudflare.

2. **Enforces identity at the edge**

   * MFA and policy can be applied consistently.

3. **Supports multi-domain**

   * easy to publish many hostnames without rearchitecting.

4. **Improves auditability**

   * Access logs and policy decisions are centralized.

5. **Fits dynamic home/edge networking**

   * IP changes do not break exposure.

## Consequences

### Positive

* Reduced public attack surface
* Strong identity enforcement for externally reachable apps
* Easy external publication of selected services

### Costs / Risks

* External dependency on Cloudflare availability
* Misconfiguration can inadvertently expose services or block operators
* Requires careful separation of:

  * external-exposed hostnames
  * internal-only hostnames

### Mitigations

* Maintain internal access methods (WireGuard) for admin operations
* Keep “break-glass” procedures documented
* Use strict allow lists and group-based access policies
* Monitor tunnel health and Access policy changes

## Security & Compliance Implications

* **Reduced attack surface** by eliminating inbound firewall exposure.

* **Centralized access logging** supports compliance evidence.

* **Least privilege** enforced via Access policies (group membership).

* **Identity integration** ensures consistent MFA enforcement.

* **Key management**:

  * Tunnel tokens/credentials must be stored in Vault.

## Implementation Notes

### Tunnel deployment pattern

* Prefer running `cloudflared` as:

  * Kubernetes workloads for app exposure, or
  * dedicated infra services for non-k8s endpoints

* Each tunnel’s credentials are stored in Vault.

### Hostname strategy

* Use explicit mapping:

  * `service.domain.tld` → tunnel → internal origin

* Support multiple domains by defining:

  * domain inventory in environment configuration
  * Access policy templates per domain

### Origin exposure

* Origins should bind to internal networks only (VLAN 20/30).
* Avoid exposing origins on DMZ directly unless strictly necessary.

### Proxmox UI access

* Proxmox UI is primarily accessed via WireGuard.
* If external access is required:

  * publish the VIP endpoint through Cloudflare Access
  * enforce strict policy and MFA

### Observability

Alert on:

* tunnel down
* high auth failure rates
* unusual access patterns

## Validation

This ADR is successful if:

* No inbound ports are required for published services.
* Operators can access allowed services externally via Access.
* Denied services are not reachable.
* Tunnel health is visible and monitored.

Suggested test cases:

* publish a non-sensitive service and verify Access gating
* attempt access from an unauthorized identity and confirm denial
* drop origin connectivity and confirm tunnel health alert

## Rollback Plan

If tunnels cause operational outage:

* disable the affected hostname routes
* restore access through internal-only methods

A permanent shift away from Cloudflare requires:

* a superseding ADR
* alternative edge/gateway design

## Follow-ups

* Define a standard naming convention for external hostnames
* Define group-based Access policy templates (admin vs user vs read-only)
* Add CI checks to prevent accidental exposure of management services
