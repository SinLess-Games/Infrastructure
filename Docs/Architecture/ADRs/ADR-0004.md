## Context

The Proxmox VE cluster is the foundational substrate for the platform. Administrative access to the Proxmox UI and API must remain available during:

* routine maintenance (patching, reboots)
* partial outages (a node down, network link down)
* emergency operations (recovery actions, storage remediation)

The desired access model is **Zero Trust**:

* The Proxmox management plane must not be exposed directly to the public Internet.
* Access must be gated via **WireGuard** (management overlay) and/or **Cloudflare Access**.

A common failure mode without HA UI access is “the node I need to fix is the node hosting the UI entrypoint.” To avoid operator lockout, the cluster requires a stable management endpoint that tolerates node failures.

## Decision

Provide a single stable Proxmox UI entrypoint using:

* **Keepalived** to host a **VRRP Virtual IP (VIP)** on **VLAN 10 (Management)**
* **HAProxy** bound to the VIP to load-balance UI/API traffic to Proxmox nodes
* **Round robin** as the load-balancing algorithm, with strict health checks

The Proxmox UI entrypoint is accessed by operators through approved paths only:

* WireGuard overlay (preferred for admin operations)
* Cloudflare Tunnel + Access (for external access when required)

## Scope

This ADR governs:

* How the Proxmox UI and API are presented to operators as a stable endpoint
* The use of Keepalived VRRP VIP + HAProxy
* The requirement that access is constrained to management plane policies (ADR-0003)

This ADR does not:

* Define the full internal DNS naming scheme for the UI VIP (covered elsewhere)
* Replace Proxmox node-level access patterns (direct node access still exists)

## Alternatives Considered

### A1) DNS round robin (multiple A records)

**Pros**

* Simple
* No additional services

**Cons**

* DNS caching causes long failover times
* No health awareness
* Operator experience is inconsistent during partial outages

### A2) “Pick a primary UI node”

**Pros**

* Simplest operational model

**Cons**

* Single point of failure
* Increases risk during maintenance or node failure

### A3) External load balancer appliance/service

**Pros**

* Robust L4/L7 features
* Often includes health checks and HA

**Cons**

* Additional hardware/service and configuration burden
* Harder to keep fully “as code” within the repo constraints

### A4) Cloudflare-only access directly to each node

**Pros**

* Strong access control at edge

**Cons**

* Still does not solve “stable internal entrypoint” problem
* Adds operational friction when Cloudflare is unavailable

## Rationale

Keepalived + HAProxy is selected because it:

1. **Provides a stable VIP**

   * Operators use one endpoint regardless of which PVE node is up.

2. **Supports fast failover**

   * VIP moves quickly when the active node fails.

3. **Supports health-aware routing**

   * HAProxy removes unhealthy nodes from rotation.

4. **Is lightweight and automatable**

   * Works well in VMs or service hosts and is easy to manage via Ansible.

5. **Aligns with Zero Trust**

   * Keeps access inside the Management plane and under overlay/tunnel policy.

## Consequences

### Positive

* Lower likelihood of operator lockout during incidents
* Consistent access endpoint for bookmarks, tooling, and documentation
* Better resilience during node maintenance

### Costs / Risks

* Additional moving parts (Keepalived + HAProxy)
* Incorrect health checks or firewall rules can cause false failures
* Proxmox UI is stateful; load balancing can occasionally affect session continuity

### Mitigations

* Use strict, accurate health checks:

  * Validate TLS endpoint responsiveness
  * Validate the node is reachable and UI/API port is open

* Consider session persistence (“stickiness”) if required for operator UX

* Restrict VIP listeners to Management plane only

* Monitor VIP status and HAProxy backend health in observability stack

## Security & Compliance Implications

* The VIP and HAProxy are part of the **Management plane**.

  * Access must be restricted via WireGuard and/or Cloudflare Access.

* Centralizing the entrypoint improves auditability:

  * access logs can be collected and retained
  * administrative access paths can be documented and controlled

* The solution avoids direct Internet exposure of management services.

## Implementation Notes

### Placement

Run Keepalived + HAProxy on **two endpoints** for high availability:

* Option A (recommended): two small VMs dedicated to mgmt LB services
* Option B: infra service endpoints hosted primarily on **pve-02** and **pve-03**

Placement requirements:

* Both endpoints must be on **VLAN 10**
* Both must have firewall rules permitting:

  * VRRP (protocol 112) between keepalived peers
  * inbound UI VIP traffic from approved sources (WireGuard clients / internal admin subnet)
  * outbound to Proxmox nodes on UI/API ports

### VIP details

* VIP resides on VLAN 10
* DNS record (internal) points to VIP

### HAProxy behavior

* Algorithm: round robin

* Backends: Proxmox nodes UI/API endpoints

* Health checks:

  * TCP connect check
  * Optional HTTPS check for `/` or a lightweight endpoint

* Optional session persistence:

  * Use if operator UX requires it
  * Prefer minimal persistence; rely on health checks first

### Observability

* Export HAProxy metrics (Prometheus exporter)
* Log VIP transitions (keepalived logs)
* Alert on:

  * VIP flapping
  * no healthy backends
  * unusually high error rates

## Validation

This ADR is successful if:

* Proxmox UI remains reachable via the VIP when any single PVE node fails
* VIP fails over within an acceptable window when the active keepalived holder is down
* HAProxy removes unreachable nodes from rotation
* Access to VIP is restricted to approved management access paths

Suggested test cases:

* reboot a PVE node while logged into UI
* stop HAProxy on the active LB node and confirm failover
* block a backend node port and confirm it leaves rotation

## Rollback Plan

If VIP + HAProxy causes instability:

* Temporarily disable HAProxy and point VIP directly to a single known-good node
* Or disable VIP and use direct node access via WireGuard for emergency operations

A permanent redesign requires a new ADR to supersede this one.
