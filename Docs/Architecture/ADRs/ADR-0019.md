## Context

The platform requires a secure, reliable management access method that:

* does not expose management services publicly
* supports administrative access from operator devices
* supports automation access from CI and orchestrators
* remains functional across changing home/edge WAN conditions
* provides a clear “management plane” distinct from workload planes

Management endpoints include:

* Proxmox UI/API/SSH
* Ceph administration endpoints
* Kubernetes control plane access (operator/admin)
* platform service administration endpoints (Vault, Authentik, GitLab, MinIO, DNS)

Constraints:

* the platform uses VLAN network segmentation (ADR-0003)
* external access uses Cloudflare Tunnel for selected services (ADR-0011)
* management surfaces must be reachable even when Cloudflare policies are changed or tunnels are degraded

Therefore, an encrypted overlay network is required for the management and automation plane.

## Decision

Adopt **WireGuard** as the management overlay network for:

* operator administrative access
* automation orchestration connectivity
* controlled access to management plane services

Rules:

* Proxmox management services are reachable only via:

  * WireGuard overlay and/or internal VLAN networks
  * optionally via Cloudflare Access if explicitly published

* No management ports are exposed directly to the public Internet.

* WireGuard peer keys and configuration secrets are stored in Vault.

## Scope

This ADR governs:

* the choice of WireGuard for the management overlay
* baseline rules for management access paths
* secret custody for peer configuration

This ADR does not:

* define the exact addressing plan (documented in network architecture)
* mandate a single topology (hub-and-spoke vs mesh) beyond operational requirements

## Alternatives Considered

### A1) OpenVPN

**Pros**

* mature

**Cons**

* heavier and more complex than WireGuard
* less ergonomic for automation

### A2) IPSec

**Pros**

* standard and widely supported

**Cons**

* higher configuration complexity
* less developer/operator friendly

### A3) Tailscale

**Pros**

* very convenient
* device identity features

**Cons**

* introduces additional external control plane
* may conflict with “self-hosted high assurance” posture

### A4) Rely on Cloudflare Access only

**Pros**

* consistent edge policy

**Cons**

* not suitable for all management access needs
* dependency on external service and tunnel health

## Rationale

WireGuard is selected because it:

1. **Provides simple, fast encrypted connectivity**

   * minimal overhead

2. **Works well for automation**

   * stable configuration and routing patterns

3. **Supports a clear management plane**

   * isolates admin access from workload networks

4. **Fits Zero Trust model**

   * explicit peer identities and least privilege routing

5. **Is robust for edge environments**

   * handles dynamic IPs and NAT well

## Consequences

### Positive

* Reliable secure access to management services
* Reduced public attack surface
* Clear operational separation between management and workloads

### Costs / Risks

* Requires key management and peer lifecycle operations
* Misconfiguration can cause lockout
* If topology is not carefully designed, routing complexity can grow

### Mitigations

* Store all keys and peer configurations in Vault
* Maintain a break-glass local console procedure
* Use standardized configuration templates and automation
* Monitor tunnel health and peer connectivity

## Security & Compliance Implications

* **Encryption in transit** for admin and automation traffic.

* **Least privilege**:

  * limit allowed routes per peer
  * restrict access to management subnets only

* **Auditability**:

  * log WireGuard endpoint access where feasible (system logs)
  * correlate with Authentik and Cloudflare logs

* **Credential hygiene**:

  * rotate keys on a schedule
  * revoke peers promptly upon device/user changes

## Implementation Notes

### Topology

Recommended starting topology:

* hub-and-spoke with a small number of “gateway” peers

Evolve to partial mesh only if necessary.

### Addressing and routing

* Allocate a dedicated overlay subnet for WireGuard.

* Define routes to:

  * management VLAN (VLAN 10)
  * selected internal services VLANs (VLAN 20) where admin access is needed

* Do not route workload pod/service networks over the management overlay by default.

### Key custody

* WireGuard private keys are stored in Vault.
* Config templates render peer configs from Vault-backed inputs.

### Automation access

* CI runners authenticate to Vault, retrieve WireGuard configuration, establish overlay connectivity, then run automation.

### Hardening

* Rate-limit and firewall the WireGuard endpoint.
* Restrict listening ports and interfaces.
* Validate kernel settings and keep systems patched.

### Observability

Monitor:

* peer handshake recency
* packet loss and throughput
* endpoint flaps

Alert on:

* management plane unreachable
* unexpected route changes

## Validation

This ADR is successful if:

* Operators can reach Proxmox UI/API over WireGuard from an authorized device.
* Automation can reach required endpoints without public exposure.
* Routes are restricted and do not allow lateral movement into workload networks.
* Key rotation and peer removal procedures work.

Suggested test cases:

* connect from operator laptop and access Proxmox VIP UI
* rotate a peer key and confirm reconnection
* revoke a peer and confirm access is blocked

## Rollback Plan

If WireGuard is unstable:

* temporarily fall back to local network-only management access
* use out-of-band console access to repair configuration

A long-term replacement requires a superseding ADR.

## Follow-ups

* Finalize WireGuard addressing plan and route policy
* Define standard peer profiles (admin laptop, CI runner, bastion)
* Implement peer lifecycle automation (add/remove/rotate)
* Document break-glass procedures and periodic access reviews
