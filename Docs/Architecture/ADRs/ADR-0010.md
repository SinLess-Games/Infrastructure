## Context

The platform must support:

* multiple domain names
* internal and external service endpoints
* strict Zero Trust posture (no public exposure of management services)
* compliance-ready controls (encryption in transit, predictable certificate rotation, auditability)

TLS certificates are required for:

* Kubernetes ingress and service gateways
* internal platform UIs (Grafana, OnCall, Kiali, GitLab, Vault UI, etc.)
* Proxmox UI access path (VIP endpoint) where applicable
* service-to-service encryption (mesh mTLS is separate, but external TLS remains required)

You explicitly require:

* **cert-manager**
* **Cloudflare** as the certificate validation mechanism
* storing certificate material in **Vault**

The platform also requires GitOps operation:

* certificate issuance configuration must be declarative and reconciled via Flux

## Decision

Adopt **cert-manager** as the Kubernetes certificate controller.

Use **Cloudflare DNS-01** validation for ACME issuance.

Store certificate material (private keys and issued certificates) in **Vault** as the long-term secret system of record.

Key design points:

* cert-manager runs in each Kubernetes cluster.

* ACME issuers use Cloudflare DNS-01 to support:

  * wildcard certificates
  * multi-domain support
  * issuance without inbound public ports

* Certificate private keys and certificates are exported to Vault.

## Scope

This ADR governs:

* how TLS certificates are requested/issued in Kubernetes
* the decision to use Cloudflare DNS-01
* the decision to store certificate material in Vault

This ADR does not:

* define the exact Vault path structure (documented in Vault ADRs and implementation docs)
* define the exact ingress controller or gateway topology
* replace service mesh mTLS (Istio) decisions (that is separate)

## Alternatives Considered

### A1) HTTP-01 validation

**Pros**

* simple

**Cons**

* requires exposing challenge endpoints
* conflicts with tunnel-first and “no inbound ports” posture
* less suitable for wildcard certs

### A2) Manual certificates (operator-managed)

**Pros**

* no automation dependencies

**Cons**

* poor rotation hygiene
* high operational toil
* weak auditability

### A3) Self-signed internal PKI only

**Pros**

* strong internal control

**Cons**

* external clients require trust distribution
* not ideal for public or semi-public endpoints

### A4) Cloudflare Origin Certificates only

**Pros**

* easy if everything is behind Cloudflare

**Cons**

* certificate trust is Cloudflare-specific
* not suitable for internal-only access paths that are not Cloudflare-mediated

## Rationale

cert-manager + Cloudflare DNS-01 is selected because it:

1. **Supports tunnel-first, no-inbound-port posture**

   * DNS-01 issuance works without exposing ACME endpoints.

2. **Supports multi-domain and wildcards cleanly**

   * reduces certificate sprawl and manual work.

3. **Fits GitOps**

   * issuers and certificates are declared as YAML and reconciled.

4. **Improves compliance and operational hygiene**

   * predictable rotation
   * auditable configuration

5. **Vault as system of record**

   * central control of secret material
   * supports disaster recovery and cross-system reuse patterns

## Consequences

### Positive

* Automated issuance and renewal
* Works with Cloudflare Tunnel-first approach
* Centralized secret custody (Vault)

### Costs / Risks

* Additional components (cert-manager, issuer config, exporters)

* Failure modes:

  * Cloudflare API token issues
  * DNS propagation delays
  * misconfigured issuer/solver selectors

* Vault integration adds complexity:

  * synchronization between Kubernetes Secrets and Vault must be correct

### Mitigations

* Use scoped Cloudflare API tokens with least privilege:

  * DNS edit only for required zones

* Monitor cert-manager events and renewal failures

* Maintain runbooks for:

  * emergency re-issue
  * token rotation
  * Vault restore and re-sync

* Keep fallback internal CA strategy documented (break-glass)

## Security & Compliance Implications

* **Encryption in transit** is enforced consistently.

* **Least privilege**:

  * Cloudflare DNS API tokens are scoped and rotated.

* **Secret custody**:

  * Vault is the authoritative store of private key material.

* **Auditability**:

  * certificate requests are visible in Kubernetes events and logs
  * Vault audit logs record secret access patterns

* **Key rotation**:

  * automated renewal reduces the risk of expired certificates

## Implementation Notes

### Issuer strategy

* Define a standard issuer hierarchy:

  * `ClusterIssuer` for shared ACME configuration
  * `Issuer` for namespace-scoped special cases

* Prefer wildcard certificates for common domains where appropriate.

### Cloudflare credentials

* Cloudflare tokens are stored in Vault.
* Tokens are injected into cert-manager namespaces using approved secret injection patterns.

### Vault storage pattern

* Each certificate is written to Vault under a defined path scheme:

  * per domain
  * per environment
  * per service

The authoritative scheme is defined in code and documented.

### Multi-domain support

* Maintain a list of supported zones/domains in environment configuration.
* Support multi-zone issuance through separate solver configurations.

### Disaster recovery

* If a cluster is rebuilt:

  * cert-manager configuration re-applies via Flux
  * Vault retains certificate material as a recovery source

### Operational monitoring

Alert on:

* certificate renewal failures
* impending expiration (threshold-based)
* Cloudflare API errors

## Validation

This ADR is successful if:

* A wildcard certificate can be issued for each supported domain.
* Certificates renew automatically without operator action.
* Certificate material is present in Vault and matches the active Kubernetes secret.
* A rebuilt cluster can re-issue or rehydrate certificates quickly.

Suggested test cases:

* issue a new wildcard cert for a test domain
* rotate the Cloudflare token and confirm issuance continues
* simulate issuer misconfiguration and confirm alerts fire

## Rollback Plan

If DNS-01 issuance becomes unreliable:

* temporarily use a reduced set of certificates and manual issuance for critical endpoints
* or switch to a fallback issuer mechanism (documented)

A permanent change requires:

* a superseding ADR
* migration plan for certificate issuance and secret custody

## Follow-ups

* Define the exact Vault path taxonomy for certificates
* Choose and document the mechanism that syncs certs into Vault (exporter/controller pattern)
* Define standard certificate naming conventions per environment
* Add policy-as-code checks requiring TLS for ingress/gateways where applicable
