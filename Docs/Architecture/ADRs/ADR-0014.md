## Context

The platform must deliver **production-grade observability** for:

* Proxmox cluster health and capacity (hosts, storage, VM inventory)
* Kubernetes clusters (dev/staging/prod) health and performance
* Workloads and platform services (Vault, Authentik, GitLab, MinIO, DNS)
* Zero Trust and compliance evidence (audit trails, retention, incident response)
* Progressive delivery (Istio + Flagger) which requires reliable telemetry signals

The required components are explicitly:

* Grafana
* Prometheus
* Mimir
* Loki
* Tempo
* Pyroscope
* Alloy
* Beyla
* Faro
* Kiali
* Grafana OnCall
* InfluxDB for Proxmox metrics

Constraints:

* GitOps-managed via FluxCD
* Multi-cluster: Development, Staging, Production
* Must support alerting and runbooks
* Must support multiple domains and SSO via Authentik

## Decision

Adopt a **Grafana-native observability architecture** with:

* **Grafana** as the primary UI for dashboards, exploration, and alerting
* **Mimir** as the long-term metrics store
* **Loki** as the log store
* **Tempo** as the trace store
* **Pyroscope** as the continuous profiling store
* **Alloy** as the standard collection/forwarding agent for metrics/logs/traces/profiles
* **Beyla** for eBPF auto-instrumentation where appropriate
* **Faro** for Real User Monitoring (RUM) for web frontends
* **Kiali** for Istio/service-mesh observability
* **Grafana OnCall** for alert routing, escalation, and incident response workflow
* **InfluxDB** as the dedicated store for **Proxmox metrics** (fed into Grafana)

Telemetry is collected in each cluster and forwarded to the appropriate backend(s) with clear retention policies and strong access controls.

## Scope

This ADR governs:

* the selection and responsibilities of the observability components
* the standard telemetry data flow across clusters and Proxmox
* the requirement to use Alloy as the default agent
* retention, alerting, and dashboard-as-code expectations

This ADR does not:

* mandate a single deployment topology (centralized vs per-cluster) beyond the baseline model described below
* define exact retention windows (they are environment-configured)
* define every dashboard, alert rule, or SLO (implemented as code over time)

## Alternatives Considered

### A1) Prometheus + Alertmanager only

**Pros**

* simpler

**Cons**

* limited long-term storage without Mimir
* logs/traces/profiles not covered
* reduced debugging capability

### A2) EFK (Elasticsearch + Fluentd/Fluent Bit + Kibana)

**Pros**

* mature log search

**Cons**

* heavier operational footprint
* not aligned with required Grafana stack

### A3) OpenTelemetry Collector + vendor-agnostic backends

**Pros**

* flexible

**Cons**

* more design work for end-to-end integration
* still requires choosing metrics/logs/traces backends

### A4) SaaS observability (Datadog/New Relic)

**Pros**

* outsourced operations

**Cons**

* cost and data residency concerns
* conflicts with self-hosted high assurance posture

## Rationale

This decision is selected because it:

1. **Matches required components and reduces integration complexity**

   * Grafana stack components are designed to work together.

2. **Supports deep debugging**

   * Metrics + logs + traces + profiles in one ecosystem.

3. **Enables progressive delivery automation**

   * Flagger can consume metrics signals; Istio provides telemetry.

4. **Improves compliance readiness**

   * controlled retention, audit logs, and evidence generation.

5. **Standardizes collection with Alloy**

   * avoids a patchwork of agents and configs.

## Architecture Overview

### Data flows (high level)

**Kubernetes clusters (dev/staging/prod):**

* **Metrics**

  * scrape local targets (kube-state-metrics, node exporters, app metrics)
  * Alloy forwards to Mimir via remote_write

* **Logs**

  * Alloy tails pod logs and forwards to Loki

* **Traces**

  * OpenTelemetry SDKs and/or mesh emit traces
  * Alloy receives OTLP and forwards to Tempo

* **Profiles**

  * Pyroscope agents (where used) or OTLP profiling flows
  * Alloy forwards to Pyroscope

* **eBPF signals**

  * Beyla runs as a daemonset and exports metrics/traces to Alloy/OTLP

* **RUM**

  * Frontends send Faro events to Faro backend
  * Faro backend forwards/enriches into Loki/Tempo/Mimir as configured

**Service mesh:**

* Istio provides telemetry and traffic control
* Kiali visualizes mesh topology, health, and traffic metrics

**Proxmox:**

* Proxmox emits metrics into **InfluxDB** (either native metric server integration or Telegraf)
* Grafana dashboards read from InfluxDB for Proxmox views

### Baseline topology: “Hub + Spoke”

* Each Kubernetes cluster runs collectors (Alloy/Beyla/Faro) locally.
* Observability backends can be deployed as:

  * **Centralized** in a “platform” namespace/cluster (preferred for production), or
  * **Per-cluster** for dev/staging initially with later consolidation.

The default posture is:

* Development and Staging may run a smaller footprint and shorter retention.
* Production runs HA backends with longer retention.

## Component Responsibilities

### Grafana

* Unified dashboards for:

  * Kubernetes cluster health
  * workload performance
  * Proxmox + Ceph health
  * security/compliance signals

* Alerting interface and rule management (with OnCall integration)

* Datasources:

  * Mimir (metrics)
  * Loki (logs)
  * Tempo (traces)
  * Pyroscope (profiles)
  * InfluxDB (Proxmox metrics)

### Prometheus

* Local scraping (where used) and alert-rule execution (where used)
* In a “Grafana stack” model, Prometheus is used as:

  * a cluster-local scrape engine
  * with remote_write into Mimir

### Mimir

* Long-term, scalable metrics store
* Receives remote_write from Alloy (and/or Prometheus)
* Retention and tenant/environment boundaries are enforced

### Loki

* Central log store
* Receives logs from Alloy
* Provides label-based query and correlation with traces

### Tempo

* Distributed trace store
* Receives OTLP traces forwarded by Alloy
* Supports trace-to-logs and trace-to-metrics linking

### Pyroscope

* Continuous profiling storage and UI integration into Grafana
* Receives profiles from agents and/or Alloy

### Alloy

* The standard telemetry agent

  * metrics scraping/forwarding
  * log shipping
  * OTLP receiver and forwarder
  * (where supported) profiling forwarder

* Runs as:

  * daemonset (node-level collection)
  * deployment (cluster-level receivers)

### Beyla

* eBPF auto-instrumentation for:

  * request metrics
  * basic tracing

* Deployed selectively (start with staging/prod after validation)

### Faro

* RUM collection for web frontends
* Used where real user performance and UX metrics matter

### Kiali

* Service mesh visibility and operational interface
* Integrates with:

  * Prometheus/Mimir metrics
  * Istio control plane

### Grafana OnCall

* Alert routing and escalation
* Supports:

  * schedules
  * rotations
  * paging rules
  * incident tracking integration

### InfluxDB (Proxmox metrics)

* Dedicated metrics store for Proxmox host + VM metrics

* Fed by:

  * Proxmox metrics server configuration (preferred), or
  * Telegraf/agent-based collection

* Grafana provides curated Proxmox dashboards against InfluxDB

## Retention and Storage Strategy

Retention is environment-driven and enforced per backend:

* Development:

  * short retention
  * smaller resource footprint

* Staging:

  * medium retention
  * mirrors production alert rules where feasible

* Production:

  * longer retention (metrics/logs/traces/profiles)
  * HA deployments
  * clear capacity planning and scaling policy

Storage backends use object storage where applicable.

Default object storage intent:

* Use **MinIO** (local S3) as an object store for components that support S3 backends.

This aligns with:

* local-first posture
* backup patterns (Velero already targets MinIO)

## Alerting Model

Alerting is standardized as:

* Rule evaluation (Prometheus and/or Grafana alerting rules as code)
* Notifications routed to **Grafana OnCall**

Baseline alert categories:

* Proxmox:

  * node down
  * high load / memory pressure
  * storage degradation

* Ceph:

  * HEALTH_WARN/ERR
  * OSD down/out
  * nearfull/full
  * slow ops

* Kubernetes:

  * API server availability
  * node not ready
  * pod crash loops
  * high error rates / latency

* Platform apps:

  * Vault sealed/unhealthy
  * Authentik unhealthy
  * GitLab degraded
  * MinIO errors

Alert rules are stored in Git and promoted through environments.

## Dashboards and Evidence-as-Code

Dashboards, alerts, and runbooks are treated as code:

* dashboards provisioned via Git-managed JSON
* alert rules stored as YAML/Jsonnet (standardized)
* runbook links embedded in alerts

Compliance/evidence expectations:

* retention policies are documented and enforced
* incident records and alert history are reviewable
* audit logs for access to observability UIs are retained

## Security & Compliance Implications

* Access is controlled via Authentik OIDC and least privilege.

* External access is gated by Cloudflare Access when used.

* Sensitive telemetry handling:

  * restrict logs containing PII
  * apply sampling and redaction patterns
  * enforce retention and access controls

* Audit logging:

  * Grafana, Vault, and Access logs support evidence generation

* Network segmentation:

  * telemetry flows stay within appropriate planes
  * no direct public exposure of backends

## Implementation Notes

### Deployment strategy

* Deploy observability stack via FluxCD as HelmReleases and Kustomizations.
* Prefer a standardized “platform-observability” bundle that is parameterized per environment.

### Multi-cluster strategy

* Each cluster runs its own collectors.
* Backends may be:

  * centralized for production (recommended), and
  * optionally per-cluster for dev/staging while iterating.

### Service discovery and labeling standards

* Enforce standard labels for correlation:

  * `app.kubernetes.io/name`
  * `app.kubernetes.io/part-of`
  * `environment`
  * `cluster`

### Correlation requirements

* Ensure trace IDs can be linked to logs where possible.
* Ensure dashboards can pivot:

  * cluster → namespace → workload → pod

### Proxmox metrics ingestion

* Configure Proxmox to emit metrics to InfluxDB.
* Standardize dashboards and alerts for Proxmox fleet health.

## Validation

This ADR is successful if:

* Each cluster produces:

  * metrics into Mimir
  * logs into Loki
  * traces into Tempo
  * profiles into Pyroscope (for selected services)

* Proxmox metrics appear in Grafana via InfluxDB.

* Alerts route correctly to Grafana OnCall.

* Progressive delivery has reliable signals:

  * Flagger canaries succeed or rollback based on metrics

Suggested test cases:

* generate a synthetic service error and confirm alerts + dashboards reflect it
* trace a request end-to-end and correlate with logs
* validate Proxmox node-down alert triggers

## Rollback Plan

If the observability stack is unstable:

* reduce scope:

  * disable Beyla and Faro first
  * keep metrics and basic logs

* fall back to cluster-local Prometheus + minimal dashboards

* iterate back to full stack once stable

A permanent redesign requires a superseding ADR.

## Follow-ups

* Define concrete retention windows per environment
* Decide centralized vs per-cluster backend topology for production and document
* Implement dashboards/alerts/runbooks as code and tie them to CI validation
* Define log redaction and PII handling policy
* Add capacity planning model for Mimir/Loki/Tempo/Pyroscope storage growth
