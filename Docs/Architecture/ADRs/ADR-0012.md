## Context

The platform requires a strong Zero Trust posture and compliance-ready controls for:

* secret storage and lifecycle management
* short-lived credentials and least privilege
* PKI and certificate/key custody
* auditability of secret access
* safe CI/CD automation without embedding secrets in Git

Secrets exist across multiple layers:

* Proxmox API credentials (Terraform/Ansible)
* Ceph client keys
* Cloudflare API tokens (DNS-01 and tunnels)
* OIDC client secrets (Authentik)
* GitLab tokens and deploy keys
* Kubernetes application secrets
* Encryption keys (backup encryption, storage encryption, etc.)

The platform is operated via GitOps and automated pipelines, therefore:

* secrets must not be stored in Git
* secrets must be accessible to automation using approved auth methods
* access must be least-privilege and auditable

You explicitly selected **Vault**.

## Decision

Adopt **HashiCorp Vault** as the system of record for:

* secrets storage
* dynamic secrets generation (where feasible)
* PKI (internal CA, intermediate CAs, certificate issuance)
* key custody (encryption keys, backup keys, service credentials)
* audit logs for secret access

Vault is treated as a **Tier-0 platform dependency**.

Vault must be:

* deployed in a highly-available manner where feasible
* backed up and recoverable
* protected by strict network and identity policies

## Scope

This ADR governs:

* selection of Vault as the secrets and PKI system
* rules for secret storage and retrieval
* baseline auth patterns for CI and clusters

This ADR does not:

* define the final Vault HA topology (integrated storage vs external backend) beyond “must be HA where feasible”
* define every secret path (path taxonomy is documented and implemented as code)

## Alternatives Considered

### A1) Kubernetes Secrets only

**Pros**

* simple
* native

**Cons**

* not sufficient as a system of record
* weaker audit story
* cross-system secret sharing becomes messy

### A2) SOPS + Git-encrypted secrets

**Pros**

* GitOps-friendly
* good audit trail

**Cons**

* key management becomes a critical problem
* rotation patterns can be heavier
* does not provide dynamic secrets

### A3) Cloud provider secrets managers

**Pros**

* managed service

**Cons**

* not aligned to on-prem-first
* multi-provider complexity
* external dependency

### A4) Passbolt/Bitwarden only

**Pros**

* good for human secrets

**Cons**

* not built for machine auth + dynamic secrets
* limited PKI patterns

## Rationale

Vault is selected because it:

1. **Centralizes secret custody**

   * one authoritative store, not many local copies

2. **Enables dynamic secrets**

   * short-lived credentials reduce blast radius

3. **Provides PKI at platform scale**

   * internal certificates and intermediates are manageable

4. **Supports audit and compliance evidence**

   * Vault audit logs + access patterns are recordable

5. **Integrates with GitOps and CI**

   * OIDC/JWT auth patterns can avoid static long-lived secrets

## Consequences

### Positive

* Eliminates secrets in Git
* Enables consistent rotation and access policy
* Improves audit evidence and incident response

### Costs / Risks

* Vault becomes a Tier-0 dependency
* Misconfiguration can break automation and access
* Requires disciplined unseal, backup, and upgrade practices

### Mitigations

* Define strict operational runbooks:

  * initialization
  * unseal / recovery
  * key rotation
  * backup and restore

* Monitor Vault health and audit logs

* Use least privilege policies with clear boundaries per environment

* Maintain break-glass procedures for emergency access (audited)

## Security & Compliance Implications

* **Audit logging**:

  * Vault audit devices are enabled and shipped to the central log store.

* **Least privilege**:

  * policies are environment-scoped and workload-scoped.

* **Credential lifecycle**:

  * prefer short-lived tokens/leases.

* **Key custody**:

  * encryption keys and certificate private keys are controlled.

* **Separation of duties**:

  * admin and operator roles are separate where feasible.

## Implementation Notes

### Vault auth methods (baseline)

Preferred auth patterns:

* **OIDC** for human UI and CLI usage
* **Kubernetes auth** for in-cluster workloads
* **JWT/OIDC (GitHub Actions)** for CI where feasible

Static tokens are allowed only for:

* initial bootstrap
* emergency break-glass

### Secret injection patterns

Kubernetes workloads should consume secrets via one of:

* Vault CSI driver
* External Secrets Operator (Vault backend)
* Vault Agent Injector

The chosen pattern(s) must:

* avoid long-lived credentials in cluster
* support rotation
* be policy-controlled

### PKI and certificate custody

* Vault manages internal PKI for:

  * internal services
  * service-to-service endpoints where mesh is not used

* External-facing certificates are managed by cert-manager + Cloudflare DNS-01, with material stored in Vault per ADR-0010.

### Path taxonomy (conceptual)

The authoritative structure is defined in code, but the intent is:

* `kv/<env>/platform/<service>/...`
* `pki/<env>/...`
* `auth/<env>/...`

### Proxmox and infrastructure credentials

* Proxmox API credentials used by Terraform/Ansible are stored in Vault.
* Ceph client keys are stored in Vault.
* Cloudflare tokens and tunnel credentials are stored in Vault.

### Multi-domain considerations

* Domain-specific secrets (API tokens, cert material) are stored per domain zone.
* Policies restrict which automation can access which domain.

## Validation

This ADR is successful if:

* No secrets are required in Git for normal operation.
* CI can authenticate to Vault using approved auth and retrieve only scoped secrets.
* Kubernetes workloads can retrieve and rotate secrets automatically.
* Vault audit logs show access patterns and are retained.

Suggested test cases:

* rotate a secret and confirm workloads update without downtime
* revoke a token and confirm access is blocked
* simulate Vault restart/failover and confirm workloads recover

## Rollback Plan

If Vault is unavailable during early rollout:

* temporarily use Kubernetes Secrets for non-critical workloads
* restore Vault from backup

A permanent change away from Vault requires:

* a superseding ADR
* migration plan for secret custody and injection patterns

## Follow-ups

* Define the final Vault HA topology and storage backend
* Choose and standardize the Kubernetes secret injection mechanism(s)
* Implement Vault policy-as-code structure and CI validation
* Document bootstrap and break-glass procedures with audit controls
