## Context

The platform requires an S3-compatible object storage service for:

* **Velero** backups (Kubernetes backups) targeting S3 APIs
* **GitLab** object storage (artifacts, uploads, packages) where feasible
* Observability backends that support object storage (Mimir/Loki/Tempo/Pyroscope, depending on deployment mode)
* General platform needs (application buckets, internal tools)

Constraints:

* Local-first posture (avoid immediate cloud dependency)
* High-assurance controls:

  * least privilege access keys
  * encryption
  * auditability
  * backups and restore drills

You explicitly selected:

* **MinIO for local S3**
* placement on **pve-02** and **pve-03** (service-hosting nodes)

MinIO becomes a critical dependency for multiple platform functions. Therefore, it must be treated as a Tier-1/Tier-0-adjacent service.

## Decision

Deploy **MinIO** as the platform’s primary **local S3-compatible object storage** with the following operating model:

1. **Placement**

   * MinIO runs primarily on **pve-02** and **pve-03**.
   * MinIO is deployed as:

     * a dedicated VM (recommended for early stability) or
     * a Kubernetes deployment once stateful operations maturity is proven.

2. **Availability and durability**

   * MinIO must be deployed with redundancy.
   * Preferred durability models (select one at implementation time):

     * MinIO distributed mode across multiple nodes/disks (recommended)
     * replication between sites (later phase)

3. **Access and exposure**

   * MinIO is not publicly exposed.
   * External access (if ever required) is only via **Cloudflare Tunnel + Access**.
   * Admin interfaces are restricted to internal access paths.

4. **Security**

   * MinIO access keys and secrets are stored in **Vault**.
   * Buckets and policies follow least privilege and environment scoping.

5. **Backups & DR**

   * MinIO itself is protected by:

     * VM backups (PBS) if VM-based
     * Velero backups if K8s-based (for config)
     * MinIO bucket replication or periodic exports where required

6. **Multi-tenancy / environment separation**

   * Bucket and/or prefix separation by environment:

     * `development-*`
     * `staging-*`
     * `production-*`

   * Separate access keys per environment and consumer service.

## Scope

This ADR governs:

* MinIO as the local S3 backend
* placement intent on pve-02/pve-03
* security and access patterns
* environment separation model

This ADR does not:

* choose the final disk layout and number of disks per MinIO node
* define the final HA topology (VM vs K8s is a maturity decision)
* define future offsite replication strategy (recommended follow-up)

## Alternatives Considered

### A1) Cloud object storage only (AWS S3 / GCS)

**Pros**

* very durable
* managed

**Cons**

* ongoing cost
* external dependency
* data residency and compliance implications

### A2) Ceph RGW

**Pros**

* integrated into Ceph
* single storage fabric

**Cons**

* operational complexity increases
* troubleshooting blast radius is larger

### A3) NFS/SMB for “object-like” use cases

**Pros**

* simple

**Cons**

* not S3 API compatible
* breaks Velero expectations

### A4) No object storage (avoid S3 dependencies)

**Pros**

* fewer services

**Cons**

* limits Velero and backends
* reduces platform flexibility

## Rationale

MinIO is selected because it:

1. **Provides S3 compatibility locally**

   * supports required tooling (Velero, GitLab).

2. **Enables local-first, hybrid-ready design**

   * can later replicate to cloud if desired.

3. **Allows strict security controls**

   * policies, keys, and audit patterns can be enforced.

4. **Supports GitOps-friendly operations**

   * configuration and consumer credentials can be declared and managed.

## Consequences

### Positive

* Enables Velero and other S3-based systems locally
* Reduces cloud reliance and cost
* Supports future hybrid replication options

### Costs / Risks

* MinIO becomes a critical dependency
* Incorrect redundancy configuration risks data loss
* Performance depends on disk/network layout

### Mitigations

* Treat MinIO as Tier-1:

  * redundant deployment
  * monitoring and alerts
  * backup and restore drills

* Enforce strict bucket policy design

* Implement capacity planning and lifecycle policies

## Security & Compliance Implications

* **Least privilege**:

  * per-service access keys, scoped to specific buckets/prefixes.

* **Encryption**:

  * TLS in transit
  * encryption at rest where feasible (disk or MinIO SSE patterns)

* **Audit evidence**:

  * MinIO access logs or audit logs must be retained.

* **Key custody**:

  * all credentials stored in Vault.

* **Data classification**:

  * define which buckets may contain sensitive or regulated data.

## Implementation Notes

### Deployment mode

Recommended maturity path:

1. VM-based MinIO distributed mode across pve-02/pve-03
2. Later evaluate Kubernetes deployment when:

   * storage and backup patterns are proven
   * operational readiness for stateful K8s is strong

### Network placement

* MinIO service endpoints bind to Infra/Services VLAN (VLAN 20).
* Avoid exposing MinIO on DMZ.

### Bucket conventions (baseline)

* `velero-development`

* `velero-staging`

* `velero-production`

* `gitlab-artifacts-production`

* `gitlab-uploads-production`

* `observability-production` (if used as object storage backend)

### Lifecycle and retention

* Use bucket lifecycle rules for:

  * cleanup of old artifacts
  * retention enforcement for backup objects

### Observability

Collect:

* MinIO health metrics
* request rates and error rates
* disk usage and growth

Alert on:

* node down
* high 5xx error rate
* low free capacity

## Validation

This ADR is successful if:

* Velero can write/read backups reliably.
* GitLab can store and retrieve artifacts from MinIO.
* Access keys are rotated without downtime.
* Restore drills confirm MinIO-backed backups are usable.

Suggested test cases:

* run a Velero backup and restore a namespace
* upload and download GitLab artifacts
* rotate MinIO access keys and confirm continued operation

## Rollback Plan

If MinIO is unstable:

* temporarily redirect critical backups to cloud object storage
* or reduce dependencies to PBS-only while stabilizing MinIO

A long-term change requires a superseding ADR.

## Follow-ups

* Decide on offsite replication strategy (MinIO replication to cloud/offsite)
* Define exact disk layout and durability model
* Define data classification and retention rules per bucket family
* Add runbooks for MinIO upgrades and incident response
