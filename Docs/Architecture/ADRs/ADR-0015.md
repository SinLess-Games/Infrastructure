## Context

Production is designed to be **on-prem first** but must be able to **expand into public cloud capacity** when local resources are constrained.

Drivers:

* workload bursts (CPU/RAM spikes, event-driven usage)
* temporary scaling needs (build pipelines, batch processing)
* resilience options (partial site failure, maintenance windows)
* cost optimization (scale down cloud when not needed)

Constraints and requirements:

* Multi-provider: AWS, GCP, Linode, and others based on price
* Zero Trust: all connectivity is encrypted, identity-gated, least privilege
* GitOps: cluster state remains declared in Git and reconciled
* Compliance: audit trails, encryption, change control
* Practicality: staged rollout; validate maturity and avoid brittle assumptions

Non-goals:

* “Teleporting pods” between clusters
* Building a single Kubernetes cluster stretched across on-prem and multiple clouds without rigorous validation

## Decision (Proposed)

Adopt a **staged hybrid burst strategy** with two phases:

### Phase 1 — Multi-Cluster Burst (recommended starting point)

Use a **separate “burst” Kubernetes cluster** in a chosen cloud provider when extra capacity is needed.

* On-prem production remains the primary cluster.
* Cloud burst cluster is brought up/down via Terraform.
* Workloads are promoted/activated in the burst cluster via GitOps (FluxCD) by environment overlays.
* Traffic is routed to the burst cluster via:

  * Cloudflare (preferred), and/or
  * a controlled L7 gateway strategy.

This phase minimizes complexity by avoiding cross-site node-level integration.

### Phase 2 — Node Pool Burst (conditional)

If maturity and network constraints are satisfied, allow **cloud worker node pools** to join the production cluster.

* The cluster remains one control plane (on-prem) with additional worker pools in cloud.
* Node pools are managed by an autoscaling mechanism and provisioned via Terraform.

This phase is higher risk and requires deeper validation (latency, routing, pod networking, storage, security).

## Scope

This ADR governs:

* the intended hybrid burst architecture for production
* how capacity is added and removed
* the staged rollout approach

This ADR does not:

* pick a single provider as “the burst provider”
* mandate cross-cloud cluster-spanning as the default
* define exact autoscaling thresholds (these are policy and workload dependent)

## Alternatives Considered

### A1) No burst (on-prem only)

**Pros**

* simplest
* fewer moving parts

**Cons**

* no capacity safety valve
* harder to handle spikes and special workloads

### A2) Full “stretched” production cluster across clouds

**Pros**

* single cluster management surface

**Cons**

* complex networking and latency constraints
* difficult storage and security model
* high operational risk

### A3) Managed Kubernetes (EKS/GKE/LKE) as primary production

**Pros**

* offloads control-plane operations

**Cons**

* weak alignment with on-prem-first intent
* increased cloud coupling and cost

### A4) Batch-only burst (CI runners / job workers only)

**Pros**

* narrow scope, lower risk

**Cons**

* does not help scale user-facing services

## Rationale

The proposed staged model is selected because it:

1. **Reduces risk early**

   * Multi-cluster burst avoids complex pod networking across sites.

2. **Preserves GitOps**

   * Workloads are activated in the burst cluster via Git changes.

3. **Supports multi-provider flexibility**

   * Terraform modules can be provider-specific while the GitOps layer stays consistent.

4. **Aligns with Zero Trust**

   * External access remains Cloudflare-first; internal control traffic uses encrypted overlays.

5. **Provides a clear maturity gate**

   * Node pool burst is optional and only adopted after proven stability.

## Consequences

### Positive

* Production gains an elastic capacity valve
* Bursts can be cost-controlled (scale down)
* Staged design avoids brittle “big bang” complexity

### Costs / Risks

* Adds operational complexity (another cluster, routing, monitoring)
* Requires careful app design for portability and statelessness
* Data gravity and storage access become constraints

### Mitigations

* Define “burst-eligible” workloads:

  * stateless services
  * cacheable APIs
  * CI/build workloads

* Keep state on-prem where feasible and route only compute to burst

* Use clear “burst policies” and runbooks

* Ensure observability coverage and cost monitoring

## Security & Compliance Implications

* **Identity and access**:

  * Cloud IAM credentials are stored in Vault with least privilege.
  * CI uses OIDC/JWT auth where possible.

* **Network security**:

  * connectivity uses encrypted tunnels/overlays
  * management access remains non-public

* **Data handling**:

  * define what data may leave on-prem
  * enforce encryption and retention

* **Audit evidence**:

  * Terraform plans/applies are recorded
  * Flux reconciliation state is recorded
  * Access logs are retained

## Implementation Notes

### Provider abstraction

Use Terraform modules with a consistent interface:

* `Terraform/Modules/Burst/aws/`
* `Terraform/Modules/Burst/gcp/`
* `Terraform/Modules/Burst/linode/`

Each module provisions:

* network primitives
* compute instances or managed node pools (if applicable)
* firewall/security groups
* DNS/tunnel integration hooks

### Connectivity

Baseline connectivity pattern:

* Cloud workloads connect to:

  * Cloudflare endpoints (public edge)
  * and/or internal services via encrypted overlay

For Phase 2 (node pool burst), validate:

* pod networking and routing
* latency tolerance
* storage and CSI behavior
* security policy consistency

### Autoscaling

Phase 1 autoscaling is “cluster-level” and externally orchestrated:

* Observability signals trigger a controlled Terraform scale action.

Phase 2 may use:

* Kubernetes Cluster Autoscaler (provider-specific), or
* external automation that modifies Terraform variables.

### Data and storage strategy

Burst workloads should:

* avoid relying on on-prem Ceph directly
* use cloud-native storage where needed, or
* use MinIO/S3-compatible patterns for shared objects

### GitOps overlays

Treat burst as a separate environment target:

* `Environments/Production/`
* `Environments/Production-Burst/` (or similar)

Workloads intended for burst are enabled via overlay switches.

## Validation

This ADR is successful if:

* A burst cluster can be created and destroyed reproducibly via code.
* Burst-eligible workloads can be activated via GitOps.
* Routing can shift traffic to burst without exposing management services.
* Observability covers burst workloads (metrics/logs/traces) consistently.

Suggested test cases:

* burst-only deployment of CI runners and batch jobs
* burst deployment of a stateless API behind Cloudflare
* scale up and scale down exercises with recorded evidence

## Rollback Plan

If burst introduces instability:

* disable burst overlays and route all traffic back to on-prem
* destroy burst resources via Terraform

A permanent change requires:

* update this ADR to ACCEPTED with final decisions, or
* supersede it with a different strategy

## Follow-ups

* Decide on the initial “burst provider” for Phase 1

* Define a burst policy document:

  * eligible workloads
  * data classification rules
  * cost thresholds and shutdown rules

* Define a routing strategy for burst traffic (Cloudflare patterns)

* Add CI pipelines to validate burst Terraform modules
