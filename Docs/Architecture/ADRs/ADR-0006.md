## Context

The platform requires three Kubernetes clusters (Development, Staging, Production) with:

* HA-capable control planes
* predictable upgrade lifecycle
* compatibility with GitOps (FluxCD)
* support for service mesh (Istio) and progressive delivery (Flagger)
* strong security posture (OIDC auth, secrets via Vault, policy controls)
* operational efficiency (reduced toil, consistent node bootstrap)

The platform is deployed **VM-first** on Proxmox:

* Packer builds templates
* Terraform provisions VM node pools
* cluster bootstrap and day-2 config are automated

You explicitly want:

* “RKE2 but the K3s version of RKE2”

Operationally, this indicates two non-negotiables:

* stay within the **Rancher ecosystem** (support matrix, tooling patterns, operational familiarity)
* prefer a **lightweight footprint** aligned with K3s characteristics

This ADR must also account for compliance and “high assurance” expectations:

* auditable upgrade process
* clear component inventory
* ability to harden the cluster baseline

## Decision

Standardize on a **Rancher K3s-family distribution** for Kubernetes clusters while keeping alignment with **Rancher ecosystem tooling** and the ability to graduate to full RKE2 when required.

Practically:

* The default runtime distribution is **K3s-family** (lightweight footprint, simplified operations)
* Cluster lifecycle, automation patterns, and ecosystem integration remain compatible with Rancher-oriented practices
* Production is allowed to move to full RKE2 if hardening or compliance requirements demand it, without rewriting the entire platform model

## Scope

This ADR governs:

* The Kubernetes distribution family and operational model
* The expectation that clusters are VM-based on Proxmox
* The compatibility requirements with GitOps, mesh, observability, and security

This ADR does not:

* Pick the final CNI (tracked in a separate ADR)
* Decide on the exact control-plane VIP / load balancing mechanism (tracked separately)
* Define the complete hardening baseline (tracked in Security ADRs and policy-as-code)

## Alternatives Considered

### A1) Full RKE2 everywhere

**Pros**

* More enterprise-hardened defaults
* Strong Rancher alignment
* Familiar Kubernetes packaging model

**Cons**

* Heavier footprint
* More moving parts than necessary for dev/staging
* Higher operational overhead for lab-scale clusters

### A2) Upstream Kubernetes via kubeadm

**Pros**

* Pure upstream model
* Maximum flexibility

**Cons**

* More lifecycle toil (bootstrap, upgrades, add-ons)
* Higher risk of “snowflake clusters” without strong discipline

### A3) Talos Linux + Kubernetes

**Pros**

* Strong security posture (immutable OS, declarative)
* Clean lifecycle management

**Cons**

* Changes the host operating model significantly
* Requires retooling bootstrap and operations

### A4) Managed Kubernetes for production (EKS/GKE/etc.)

**Pros**

* Offloads control-plane operations

**Cons**

* Conflicts with on-prem-first and GitOps promotion model
* Adds cost and cloud coupling
* Multi-provider becomes complex quickly

## Rationale

This decision is selected because it:

1. **Reduces operational overhead**

   * K3s-family distributions are lighter and generally simpler to operate.

2. **Maintains ecosystem alignment**

   * Stays within Rancher-adjacent patterns and reduces integration surprises.

3. **Fits VM-first automation**

   * Works cleanly with Proxmox templates, Terraform-provisioned nodes, and automated bootstrap.

4. **Keeps an upgrade path to heavier hardening**

   * If production requires stricter defaults or compliance posture, full RKE2 remains an upgrade path.

5. **Supports the required add-ons**

   * Istio + Flagger canaries, FluxCD GitOps, Vault integration, and Grafana observability remain viable.

## Consequences

### Positive

* Faster cluster provisioning and rebuilds
* Lower resource footprint (especially useful in dev/staging)
* Simpler baseline operations

### Costs / Risks

* Some enterprise/security features may require additional configuration compared to full RKE2.
* Must validate compatibility and support for:

  * Istio (sidecar injection, CNI considerations)
  * OIDC auth integration
  * admission policy tooling
  * CSI drivers and storage integrations

### Mitigations

* Treat production as “strict mode”:

  * additional hardening
  * stricter policy-as-code
  * more conservative upgrade cadence

* Maintain documented conformance tests and “cluster readiness checks” in CI

## Security & Compliance Implications

* **Component inventory and SBOM posture**

  * The distribution’s packaged components must be documented and versioned.

* **Hardening**

  * Baseline hardening is implemented via:

    * OS baseline (Ansible)
    * cluster config defaults
    * admission policies
    * network policies

* **Audit logging**

  * Kubernetes audit logs are enabled and centrally retained.

* **Change control**

  * Cluster version upgrades must be PR-driven with documented testing.

## Implementation Notes

### Bootstrap model (VM-first)

* Packer builds a base node template
* Terraform provisions node pools per environment
* Node bootstrap installs the distribution and joins nodes automatically

### Environment strategy

* Development:

  * prioritize speed and rebuildability

* Staging:

  * mirrors production policies and version windows as closely as practical

* Production:

  * strict policy enforcement
  * conservative upgrades
  * may move to full RKE2 if hardening demands it

### Integration requirements

This distro choice must support:

* FluxCD reconciliation
* Istio + Flagger
* Vault secret injection patterns
* cert-manager for certificates
* observability agents (Alloy/Beyla) and telemetry endpoints

### Upgrade policy (baseline)

* Maintain a tested version window:

  * N (current)
  * N-1 (previous)

* Upgrades are executed:

  * in dev first
  * then staging
  * then production

* Each upgrade has:

  * pre-flight checks
  * post-flight checks
  * rollback plan

## Validation

This ADR is successful if:

* All three clusters can be created reproducibly from code
* Node replacement is automated (new VM joins correctly)
* GitOps reconciliation remains stable during upgrades
* Istio + Flagger canaries function correctly
* OIDC auth works for cluster access and UIs

Suggested test cases:

* rebuild a dev cluster from scratch and verify platform bootstrap completes
* run a staged upgrade dev → staging → prod
* run an Istio/Flagger canary and verify automatic promotion/rollback

## Rollback Plan

If the chosen K3s-family distribution fails to meet production requirements:

* Promote a controlled migration to full RKE2 in production only.
* Preserve:

  * GitOps structure
  * observability stack
  * secret management patterns

A major change to a different distro family requires:

* a superseding ADR
* a migration plan
