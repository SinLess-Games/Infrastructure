## Context

The platform must enforce a Zero Trust access model:

* No public exposure of management services
* Strong authentication for all administrative and sensitive service access
* Centralized identity, group membership, and role mapping
* Compliance-ready change control and auditability
* Support for multiple domains and consistent SSO across them

Key surfaces requiring authentication and authorization:

* **Proxmox VE** (UI/API)
* **Kubernetes** (kubectl access, API authentication where applicable)
* **Grafana** and observability UIs (including OnCall)
* **GitLab** (developer and operator workflows)
* **Vault UI/API**
* **Kiali**, **Istio dashboards**, other platform UIs
* Potential future workloads and SaaS endpoints

Requirements:

* SSO must work both for internal-only access (WireGuard) and for external access via Cloudflare Access.
* RBAC must be consistent and enforceable across systems.
* The identity provider must support:

  * OIDC
  * group claims
  * MFA
  * audit/event visibility
  * integration with reverse proxies/access gateways

You explicitly selected **Authentik**.

## Decision

Adopt **Authentik** as the central Identity Provider (IdP) for the platform.

Authentik will provide:

* OIDC for SSO across platform applications
* group-based access control signals (claims)
* MFA enforcement and policy
* centralized user lifecycle management

Integrations:

* **Proxmox VE:** OIDC realm configured to Authentik; roles mapped from group claims.
* **Kubernetes:** OIDC authentication for human access and UI access patterns where applicable; RBAC bindings mapped to Authentik group claims.
* **Grafana / OnCall / Kiali / GitLab / Vault UI:** OIDC SSO via Authentik.
* **Cloudflare Access:** can optionally use Authentik as the upstream IdP, or Cloudflare as the IdP with Authentik used for app-level SSO. (The preferred model is documented in implementation as code.)

## Scope

This ADR governs:

* choice of Authentik as the central IdP
* use of OIDC for SSO and group claims
* requirement that platform services avoid local password auth where feasible

This ADR does not:

* define the final “Access gateway of record” (Cloudflare Access vs internal proxy) beyond acknowledging Cloudflare is used for external access
* replace application-specific configuration runbooks (those live in ops docs)

## Alternatives Considered

### A1) Keycloak

**Pros**

* very mature, enterprise-grade
* strong federation options

**Cons**

* heavier operational footprint
* more complex than needed for the platform’s initial scope

### A2) Auth0 / Okta (managed IdP)

**Pros**

* outsourced availability and lifecycle

**Cons**

* introduces external dependency and cost
* reduces control for “self-hosted high assurance” posture

### A3) Cloudflare Access only (no internal IdP)

**Pros**

* simple for edge-gated apps

**Cons**

* not sufficient for internal-only apps
* limited cross-app RBAC unification
* makes internal SSO patterns more fragile

### A4) Local accounts per service

**Pros**

* trivial setup

**Cons**

* inconsistent authentication
* hard to audit and enforce MFA
* poor compliance posture

## Rationale

Authentik is selected because it:

1. **Provides OIDC + group claims + MFA in one self-hosted system**

   * central policy control

2. **Improves auditability and access governance**

   * consistent auth decisions

3. **Scales to multiple domains**

   * consistent identity across services and hostnames

4. **Reduces credential sprawl**

   * fewer local password databases

5. **Aligns with Zero Trust posture**

   * easier to enforce least privilege and strong authentication

## Consequences

### Positive

* One identity plane for platform services
* Consistent RBAC mapping and access patterns
* MFA can be enforced centrally

### Costs / Risks

* Authentik becomes a Tier-0 dependency for SSO flows
* Misconfiguration can lock out operators
* Requires backup/restore and HA considerations

### Mitigations

* Provide a documented break-glass path:

  * limited local admin accounts where absolutely required
  * stored securely and audited

* Backup Authentik configuration and database

* Deploy Authentik with redundancy where feasible

* Monitor Authentik health and login failures

## Security & Compliance Implications

* **MFA enforcement** becomes centrally manageable.

* **Least privilege** is supported by group-to-role mapping.

* **Audit evidence**:

  * Authentik events + service access logs support audits.

* **Credential reduction**:

  * minimizes password reuse and unmanaged credential growth.

* **Break-glass controls** must be documented and periodically tested.

## Implementation Notes

### Deployment placement

* Authentik runs as a platform service (likely on the Infra/Services plane).
* External exposure is via Cloudflare Tunnel + Access (when needed).

### Claims model

* Standardize on group claim keys (example intent):

  * `groups` claim with values such as:

    * `platform-admin`
    * `platform-operator`
    * `platform-readonly`
    * `dev-team`

The authoritative naming is defined in code.

### Proxmox mapping

* Configure a Proxmox OIDC realm.
* Map Authentik groups to Proxmox roles and permissions.

### Kubernetes mapping

* Configure OIDC authentication for user access.
* Map group claims to Kubernetes RBAC:

  * ClusterRoleBindings and RoleBindings

### Multi-domain support

* Authentik application definitions must support multiple hostnames.
* Certificates are issued via cert-manager + Cloudflare and stored in Vault per certificate strategy ADR.

### Secrets

* All OIDC client secrets are stored in Vault.

## Validation

This ADR is successful if:

* Operators can authenticate to:

  * Proxmox UI via OIDC
  * Grafana and OnCall via OIDC
  * GitLab via OIDC

* Group membership changes in Authentik correctly update access permissions.

* MFA is enforceable and tested.

* Break-glass procedure exists and works.

Suggested test cases:

* remove a user from `platform-admin` and confirm loss of admin access
* enforce MFA and validate login workflow
* simulate Authentik outage and confirm break-glass access methods

## Rollback Plan

If Authentik fails operationally during early rollout:

* temporarily use local accounts for critical services
* restore Authentik from backup

A permanent IdP migration requires:

* a superseding ADR
* staged migration plan (client secrets, redirects, role mappings)

## Follow-ups

* Define the standard group taxonomy and role mapping tables in code
* Define HA pattern for Authentik deployment
* Document break-glass operational procedure with audit controls
